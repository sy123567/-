## 1. 天气数据源（已确定并验证）

- 服务商：**华风爱科 / 中国天气开放平台**（`platform.weathercn.com`，AccuWeather 风格接口），**不是**和风天气 QWeather。
- Base URL：`https://openapi.weathercn.com`
- 鉴权：URL 参数 `apikey=<KEY>`；请求路径都带 `.json` 后缀。
- 免费档：QPS 5/秒，日调用 500 次，30 天试用（可续期）。
- **API Key 放环境变量 `WEATHER_KEY`，绝不提交 Git；聊天里明文出现过的 Key 建议在控制台重置。**

### 关键调用规律（两步）
1. 经纬度 → LocationKey：
   ```
   GET /locations/v1/cities/geoposition/search.json?apikey=KEY&q=纬度,经度&language=zh-cn
   ```
   返回 JSON 顶层 `Key` 就是 LocationKey。
2. 用 LocationKey 查：
   - 预警：`GET /alerts/v1/{Key}.json?apikey=KEY&language=zh-cn`（无灾害时返回 `[]`）
   - 逐日预报：`GET /forecasts/v1/daily/1day/{Key}.json?apikey=KEY&language=zh-cn&details=true`
   - 实况：`GET /currentconditions/v1/{Key}.json?...`
   - 空气质量：`GET /airquality/v1/global/observations/{Key}.json?...`

### 已验证的样例
| 城市 | 经纬度 | LocationKey |
|---|---|---|
| 上海(黄浦) | 31.24,121.49 | 60782 |
| 桂林(象山) | 25.274,110.290 | 2332667 |
| 南宁(青秀) | 22.817,108.366 | 2332685 |
| 阳朔 | 24.778,110.496 | 58403 |
| 北京(示例) | — | 101924 |

- 预警 JSON 关键字段：`Description.Localized`（如"雷电黄色预警"）、`Category`(SEVERE)、`Level`(黄色)、`AlarmLevel`(Red/Orange/Yellow/Blue)、`Type`(雷电)、`Area[].StartTime/EndTime`。
- 预报降水字段：`DailyForecasts[0].Day.HasPrecipitation`(bool)、`Day.PrecipitationProbability`、`Headline.Text`。

### 第 3 步：真实天气接入（已完成并推送）✅
在 `EventIngestionService`（或 monitor 包）实现，核心逻辑：
- 新增 `WeatherClient`（`@Component`）：`locationKey(lat,lon)` / `alerts(key)` / `dailyForecast(key)`，用 `RestTemplate`+`ObjectMapper`，失败返回 null 降级。读 `@Value("${app.weather.key}")`、`host`；`enabled()` 判断是否配了 Key。
- `ExternalEventRepository` 加去重方法：
  ```java
  boolean existsBySourceAndPlaceNameAndStartTime(String source, String placeName, java.time.LocalDateTime startTime);
  ```
- `EventIngestionService.ingestWeatherForTrip(tripId)`：遍历行程节点 → 查 LocationKey →
  - 有 `alerts` → 生成 WEATHER 事件（`source=weathercn-alert`，标题取 `Description.Localized`，severity 按 `AlarmLevel` 映射：Red=CRITICAL/Orange=HIGH/Yellow=MEDIUM/Blue=LOW）；
  - 无 alerts → 看预报，`HasPrecipitation` 或 `Headline.Text` 含"雨/雪/暴/雷" → 生成 MEDIUM 事件（`source=weathercn-forecast`）；
  - 用 `fill()` 填公共字段（placeName/经纬度/radiusKm/起止时间/source），`save()` 去重后存库。
- `EventController` 加接口：
  ```java
  @PostMapping("/trips/{id}/events/weather")
  public List<ExternalEvent> weather(@PathVariable Long id) { return s.ingestWeatherForTrip(id); }
  ```

**踩过的坑（已修）**：
1. 初期返回 `[]`：`WEATHER_KEY` 没配进 Run Config 或没重启（`enabled()` 为 false 直接返回空）。
2. 500 `Data too long for column 'description'`：早期把整段 JSON 塞进 `description`(VARCHAR255)。**已改为存简短摘要**（预警存 `Description.Localized`，预报存 `Headline.Text`）。如需存长文本可给实体字段加 `@Column(columnDefinition="TEXT")`。
3. 行程日期在未来（如 7-24）时天气 API 无数据 → 演示要把节点日期设成今天附近，或该行程状态设 `ONGOING`。

**验证方式（广西示例，当天有雨+预警，效果最好）**：
```
POST /api/trips?groupId=1   body: {"title":"广西桂林三日游","status":"ONGOING","startDate":"今天","endDate":"+2天","totalBudget":3000}
POST /api/trips/{id}/nodes  body: {"name":"象鼻山","placeName":"桂林象鼻山","latitude":25.274,"longitude":110.290,"nodeType":"ATTRACTION","plannedStart":"今天T09:00:00","plannedEnd":"今天T11:00:00","cost":60,"sequenceOrder":1,"status":"PLANNED"}
POST /api/trips/{id}/events/weather   → 返回"雷电黄色预警"事件
GET  /api/events   → 验证
```

## 3. 待完成（第 4 步及以后）⬜ —— 下一个会话从这里继续

监测与应变的完整闭环：
```
事件接入(✅) → 影响匹配 → 风险评分 → 消息通知 → 自动重规划 → 协同投票 → 变更记录
```

### 第 4 步：定时轮询 + 串联影响匹配 ⬜（下一步）
1. **定时轮询**（把"手动拉"变"自动拉"）：
   - `TripAdaptiveApplication` 加 `@EnableScheduling`。
   - 新建 `WeatherPollingJob`（`@Component`），`@Scheduled(initialDelay=10_000, fixedDelay=600_000)`，遍历 `trips.all()`，对 `status==ONGOING` 的行程调 `ingestWeatherForTrip(id)`。去重已在 `save()` 保证，反复轮询不重复插。
   - 验证：重启看控制台每 10 分钟触发；`GET /api/events` 有事件且不重复。测试期可临时把 `fixedDelay` 改 `60_000`。
2. **串联影响匹配**：
   - 先读 `ImpactController` / `ImpactMatchingService` 确认现有接口与方法签名（**下个会话第一件事：读这两个文件**）。
   - Swagger 手动触发一次影响匹配（对某行程），查 `impact_assessment` 表验证"哪个节点受哪条事件影响"。
   - 打通后，可在 `ingestWeatherForTrip` 末尾自动调用 `impactMatchingService.match(tripId)`（注入进来）。

### 第 5 步：风险评分（`RiskScoringService`）⬜
- 结合事件 severity + 受影响节点重要性算风险分，超阈值标记预警。
- 已有视图 `v_active_risk_snapshot`（在复数表体系；后端用单数表，需按后端实体核对）。

### 第 6 步：消息通知（`NotificationService`）⬜
- 给受影响行程的成员生成 `notification` 记录。
- 前端先用**轮询**未读通知显示红点；项目已有 `WebSocketConfig`，进阶可改实时推送。

### 第 7 步：自动重规划 + 协同投票 + 变更记录 ⬜
- `ReplanningService` 生成 `alternative_plan`；`VotingService` 记 `plan_vote`；通过后写 `node_change`/`change_log`。
- 已有 `ReplanController`/`VoteController`/`ChangeLogController`，逐个用 Swagger 验证后串联。

### 测试与提交（每步都做）
- 单元测试：JUnit5 + Mockito，mock `WeatherClient`/repository，测"有预警→出事件、无预警看预报、重复不重复插、AlarmLevel→severity 映射"。参考 `src/test/java/.../ImpactMatchingServiceTest.java`。
- 接口测试：`@SpringBootTest`+`MockMvc` 调各接口断言 200 + 数据变化。
- `mvn test` 全绿后提交。
- 分支策略：`feat/monitor-*` → PR base 选 `dev` → 队友 review → 合并；`dev` 阶段性合 `main`。不直接 push `main`，不 `git add .`。
