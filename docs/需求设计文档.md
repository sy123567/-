## 1. 引言

### 1.1 编写目的
本文档定义"智能旅游平台"的完整需求与概要设计，作为开发、测试、验收与答辩的依据，供项目组全体成员、指导教师及评审使用。

### 1.2 项目背景
普通旅行类应用通常只负责"生成一次行程"，一旦出行过程中遇到暴雨、道路施工、交通管制、景点闭馆、大型活动等外部事件，行程即失效，用户只能手动重排。本平台强调**持续规划**与**抗中断**能力：覆盖出行的完整生命周期，形成
`成员约束 → 初始计划 → 外部事件 → 影响分析 → 替代方案 → 集体确认 → 新计划`
的动态闭环。

### 1.3 项目范围
- 覆盖：多人组队、出发前共同规划、行程中事件监测、影响分析、自动重规划、多人投票决策、变更落地、实时同步、社交与增值功能。
- 不覆盖：真实支付结算、机票/酒店真实预订下单（仅做占位与费用记录）。

### 1.4 术语表

| 术语 | 说明 |
| --- | --- |
| 出行小组 (TravelGroup) | 一次出行的成员集合 |
| 成员约束 (MemberConstraint) | 单个成员的日期/预算/必访地/体力/饮食/无障碍等限制 |
| 行程节点 (ItineraryNode) | 行程中的一个点：景点/餐饮/住宿/交通 |
| 外部事件 (ExternalEvent) | 天气/交通/施工/闭馆/大型活动等影响因素 |
| 影响评估 (ImpactAssessment) | 事件与行程节点的时空关联结果 |
| 替代方案 (AlternativePlan) | 中断后自动生成的可选新计划 |
| 房间码 (RoomCode) | 用于加入出行小组的短码 |

## 2. 总体说明

### 2.1 产品定位
一个"会持续帮你重新规划"的多人出行协作平台：不仅生成行程，更能在行程被外部事件打断时，自动评估影响、生成多套替代方案、由全员投票确认后自动落地。

### 2.2 核心创新
1. **持续规划闭环**：从一次性规划升级为"监测—分析—重规划—确认"的动态循环。
2. **抗中断能力**：外部事件驱动的自动重规划，输出"最低额外成本 / 最少延误 / 最少改动"多目标方案。
3. **多人协同决策**：方案对比 + 投票 + 原子化落地，兼顾集体意见与数据一致性。
4. **算法为主、AI 为辅**：核心为约束满足与多目标优化，自然语言解析可选接入 AI，不强依赖大模型。

### 2.3 动态闭环总览
```
用户注册/组队(好友·房间码)
        │
        ▼
成员填写约束 ──▶ 初始方案生成(约束求解) ──▶ 全员查看行程
        │                                        │
        │                                        ▼
        │                              行程中事件接入(天气/交通/闭馆…)
        │                                        │
        │                                        ▼
        │                              影响匹配(时空关联) ──▶ 风险评分
        │                                        │
        │                                        ▼
        │                              自动重规划(3 套替代方案)
        │                                        │
        │                                        ▼
        │                              多人投票 ──▶ 过半通过
        │                                        │
        │                                        ▼
        └──────────── 变更落地(更新行程+变更记录) ─┘  (回到监测，持续循环)
```

## 3. 系统架构设计

### 3.1 分层架构
```
┌───────────────────────────────────────────────┐
│  客户端 (Web / App)                             │
│  REST 调用 + WebSocket(STOMP) 订阅               │
└───────────────────────────────────────────────┘
                    │ HTTP / WS
┌───────────────────────────────────────────────┐
│  Controller 层  (REST API + 参数校验 + OpenAPI) │
├───────────────────────────────────────────────┤
│  Service 层     (业务逻辑 / 算法 / 事务)          │
│  规划 · 影响匹配 · 风险 · 重规划 · 投票 · 通知     │
├───────────────────────────────────────────────┤
│  Repository 层  (Spring Data JPA)               │
├───────────────────────────────────────────────┤
│  Domain 层      (JPA 实体 / 领域模型)             │
└───────────────────────────────────────────────┘
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
   关系数据库(H2/MySQL)     外部数据源(天气/交通/地图 API)
```

### 3.2 技术选型

| 层次 | 技术 | 说明 |
| --- | --- | --- |
| Web 框架 | Spring Boot 3.3.x | REST + 依赖注入 |
| 持久化 | Spring Data JPA + Hibernate | ORM |
| 数据库 | H2(内存，开发) / MySQL(生产) | 可切换 |
| 实时通信 | Spring WebSocket + STOMP | 行程实时同步 |
| 接口文档 | springdoc-openapi (Swagger UI) | 在线调试 |
| 校验 | Jakarta Bean Validation | 请求参数校验 |
| 鉴权 | Spring Security + JWT | 用户认证（模块 1） |
| 监控 | Spring Boot Actuator | 健康检查 |

### 3.3 部署形态
- 开发：`mvn spring-boot:run`，H2 内存库，Swagger UI 在线调试。
- 生产：打包 `jar` + MySQL，Nginx 反向代理，WebSocket 走 wss。

## 5. 功能需求（20 个模块）

> 每个模块给出：功能描述、功能点、主要接口、涉及实体、异常/边界。难度标记：🟢 简单 CRUD / 🟡 中等 / 🔴 难点。

### 模块 1｜用户注册 / 登录 / 鉴权 🟢
- **描述**：账号体系与身份认证，是其他模块的前提。
- **功能点**：注册、登录、登出、修改密码、个人资料维护、JWT 签发与校验。
- **接口**：`POST /api/auth/register`、`POST /api/auth/login`、`GET /api/users/me`、`PUT /api/users/me`。
- **实体**：User（密码加密存储）。
- **异常/边界**：邮箱/手机号唯一；密码强度校验；Token 过期与刷新。

### 模块 2｜好友系统 🟢
- **描述**：用户间建立好友关系，作为组队入口之一。
- **功能点**：搜索用户、发送好友申请、同意/拒绝、好友列表、删除好友。
- **接口**：`POST /api/friends/requests`、`POST /api/friends/requests/{id}/accept`、`POST /api/friends/requests/{id}/reject`、`GET /api/friends`、`DELETE /api/friends/{userId}`。
- **实体**：Friendship（requester、addressee、status: PENDING/ACCEPTED/REJECTED）。
- **异常/边界**：不能加自己；重复申请去重；双向关系一致性。

### 模块 3｜房间码组队 🟡
- **描述**：通过房间码快速加入出行小组。
- **功能点**：创建小组自动生成房间码、按码加入、房间码有效期与人数上限、群主转让、移除成员。
- **接口**：`POST /api/groups`（返回 roomCode）、`POST /api/groups/join`（body: roomCode）、`POST /api/groups/{id}/transfer-owner`、`DELETE /api/groups/{id}/members/{memberId}`。
- **实体**：TravelGroup（roomCode、roomCodeExpireAt、memberLimit）、GroupMember。
- **异常/边界**：房间码唯一且不易碰撞；过期/满员拒绝；重复加入幂等。

### 模块 4｜成员约束管理 🟢
- **描述**：采集每个成员的出行限制，作为规划输入。
- **功能点**：设置/更新可用日期区间、预算上限、必访地点、体力等级、饮食需求、无障碍需求。
- **接口**：`PUT /api/groups/{id}/members/{memberId}/constraint`、`GET /api/groups/{id}/members`。
- **实体**：MemberConstraint。
- **异常/边界**：日期区间合法性；必访地点去重。

### 模块 5｜出发前共同规划 / 初始方案生成 🔴
- **描述**：综合全体成员约束，求解生成一份初始行程并解释取舍。
- **功能点**：日期交集计算、预算按最低上限封顶、必访地点覆盖、节点时间排布、路线生成、超预算裁剪、取舍说明。
- **接口**：`POST /api/groups/{groupId}/plan`。
- **实体**：Trip、ItineraryNode、Route。
- **异常/边界**：日期无交集 → 冲突提示；预算不可行 → 裁剪或报错；无必访地点 → 使用默认模板。
- **算法**：见 §8.1。

### 模块 6｜行程与节点管理 🟢
- **描述**：对行程节点进行手动增删改与排序。
- **功能点**：新增/编辑/删除节点、调整顺序（拖拽排序）、查看行程时间线、查看路线。
- **接口**：`GET/POST /api/trips/{id}/nodes`、`PUT /api/trips/{id}/nodes/{nodeId}`、`DELETE .../{nodeId}`、`PUT /api/trips/{id}/nodes/reorder`、`GET /api/trips/{id}/routes`。
- **实体**：Trip、ItineraryNode、Route。
- **异常/边界**：排序序号连续性；删除节点级联更新路线。

### 模块 7｜地图路线与导航 🔴
- **描述**：节点间真实路网距离/时长与路线展示。
- **功能点**：接入地图 API（或图算法）计算距离/时长/路线、多种交通方式、地图可视化数据。
- **接口**：`GET /api/trips/{id}/routes`、`POST /api/routes/plan`。
- **实体**：Route（distanceKm、durationMinutes、transportMode、polyline）。
- **异常/边界**：地图 API 限流/失败降级为直线估算；坐标缺失处理。
- **算法/集成**：见 §8.5。

### 模块 8｜预算与费用管理 🟢
- **描述**：行程预算与实际花费记录。
- **功能点**：设置总预算、按节点记录费用、汇总统计、预算超支预警。
- **接口**：`GET /api/trips/{id}/budget`、`POST /api/trips/{id}/expenses`。
- **实体**：Trip.totalBudget、ItineraryNode.cost、Expense（可选）。
- **异常/边界**：金额精度（BigDecimal）；超支提示。

### 模块 9｜天气 / 交通 / 城市事件接入 🟡
- **描述**：持续获取外部事件并入库。
- **功能点**：真实第三方 API 接入（天气/交通/公告）、Mock 事件生成、事件列表与筛选、活跃事件查询。
- **接口**：`POST /api/events`、`GET /api/events`、`GET /api/events/active`、`POST /api/trips/{tripId}/events/mock`。
- **实体**：ExternalEvent（type、地点、半径、时间窗、severity、source）。
- **异常/边界**：外部 API 限流/超时/字段缺失容错；时区处理。

### 模块 10｜事件—行程影响匹配 🔴
- **描述**：判断哪些事件影响哪些行程节点。
- **功能点**：地理距离（Haversine）+ 时间窗重叠判定、影响分级、标记受影响节点、生成影响评估。
- **接口**：`POST /api/trips/{tripId}/assess`、`GET /api/trips/{tripId}/impacts`。
- **实体**：ImpactAssessment。
- **异常/边界**：坐标/时间缺失跳过；同一节点多事件叠加取最严重。
- **算法**：见 §8.2。

### 模块 11｜风险评分与预警 🔴
- **描述**：把模糊风险量化为 0–100 分并给出预警。
- **功能点**：多因子加权（严重度、受影响节点数、成本暴露、剩余缓冲时间）、行程聚合风险报告、分级预警。
- **接口**：`GET /api/trips/{tripId}/risk`。
- **实体**：ImpactAssessment.riskScore、风险报告 DTO。
- **异常/边界**：无影响时风险为 0；权重可配置。
- **算法**：见 §8.3。

### 模块 12｜自动重规划 🔴（核心最难）
- **描述**：中断后生成多套可行替代方案。
- **功能点**：基于影响评估，按三种策略（最低额外成本 / 最少延误 / 最少改动）生成方案；每套含具体节点变更（改期/替换/删除/新增）、增量成本、延误、改动数、文字说明。
- **接口**：`POST /api/trips/{tripId}/replan`、`GET /api/trips/{tripId}/plans`、`GET /api/plans/{planId}`。
- **实体**：AlternativePlan、NodeChange。
- **异常/边界**：无可行替代 → 提示人工介入；替换候选需真实可行。
- **算法**：见 §8.4。

### 模块 13｜多人协同决策 / 投票 🔴
- **描述**：对替代方案发起投票并原子化落地。
- **功能点**：发起投票、成员投票（赞成/反对/弃权）、实时票数、计票、过半通过后原子应用变更并驳回其余方案。
- **接口**：`POST /api/plans/{planId}/start-voting`、`POST /api/plans/{planId}/votes`、`GET /api/plans/{planId}/votes`、`POST /api/plans/{planId}/tally`。
- **实体**：PlanVote、AlternativePlan（状态机 PROPOSED→VOTING→ACCEPTED/REJECTED）。
- **异常/边界**：一人一票（幂等 upsert）；并发计票需事务与锁；重复应用防护。
- **设计**：见 §8.6。

### 模块 14｜变更记录 🟢
- **描述**：记录每次采纳方案的变更与成本/退款信息。
- **功能点**：变更列表、额外成本、退款/改签截止时间、关联方案。
- **接口**：`GET /api/trips/{tripId}/changelogs`。
- **实体**：ChangeLog。
- **异常/边界**：只读；退款截止时间按规则计算。

### 模块 15｜消息通知 🟡
- **描述**：实时 + 站内信双通道通知。
- **功能点**：WebSocket 广播（事件/影响/新方案/投票/采纳）、站内信、未读计数、按小组订阅。
- **接口**：WS `/ws` 订阅 `/topic/trips/{tripId}`；REST `GET /api/notifications`、`POST /api/notifications/{id}/read`。
- **实体**：Notification（可选持久化）。
- **异常/边界**：订阅鉴权；离线消息补发；去重。

### 模块 16｜AA 分账与结算 🟡
- **描述**：出行费用拆分与成员间清算。
- **功能点**：录入共同消费、按人/按比例拆分、生成成员账单、最小转账次数清算建议。
- **接口**：`POST /api/trips/{id}/bills`、`GET /api/trips/{id}/settlement`。
- **实体**：Bill、BillShare。
- **异常/边界**：金额精度与四舍五入平账；最小转账算法。

### 模块 17｜讨论区 / 评论 🟡
- **描述**：行程与方案的成员讨论。
- **功能点**：发帖/评论、@成员、点赞、分页、针对方案的讨论串。
- **接口**：`POST /api/trips/{id}/comments`、`GET /api/trips/{id}/comments`、`POST /api/plans/{id}/comments`。
- **实体**：Comment（targetType、targetId、mentions）。
- **异常/边界**：敏感词/长度校验；@ 触发通知。

### 模块 18｜数据统计与后台 🟡
- **描述**：运营与出行数据看板 + 管理端。
- **功能点**：出行数量/预算分布/事件命中率/重规划采纳率等聚合统计、图表数据、用户与事件源管理。
- **接口**：`GET /api/admin/stats`、`GET /api/admin/users`、`GET /api/admin/events`。
- **实体**：聚合视图 / 统计 DTO。
- **异常/边界**：管理员权限校验；大数据量分页与缓存。

### 模块 19｜攻略社区（浏览 / 分类搜索 / 筛选）🟡
- **描述**：用户发布与浏览旅游攻略，攻略关联一条可复用的行程模板，是内容生态与"攻略纳用"的来源。
- **功能点**：
  - 发布攻略（图文 + 关联行程模板）、编辑、下架。
  - 分类：目的地/城市、主题（美食/亲子/徒步/文化/穷游…）、天数、人均预算、季节。
  - 分类搜索 + 多条件筛选（城市 / 价格区间 / 天数 / 主题 / 评分）+ 关键词搜索。
  - 排序（最新/最热/评分/收藏数）、分页、瀑布流浏览。
  - 收藏、点赞、浏览量、评分/评价。
- **接口**：`GET /api/guides`（filter/sort/page 参数）、`GET /api/guides/{id}`、`POST /api/guides`、`PUT /api/guides/{id}`、`POST /api/guides/{id}/favorite`、`POST /api/guides/{id}/rate`。
- **实体**：TravelGuide（title、cover、author、city、tags、days、perCapitaBudget、season、rating、favoriteCount、viewCount、status）、GuideFavorite、GuideRating、GuideTemplate（模板行程，节点用相对天/时表示）、GuideTemplateNode。
- **异常/边界**：多条件组合筛选查询性能（建议 JPA Specification/QueryDSL + 索引）；热度/评分排序；下架攻略不参与检索。
- **难度**：查询与排序为主，属 🟡；若做复杂检索/推荐可上探。

### 模块 20｜攻略纳用（可编辑）🔴
- **描述**：把攻略的行程模板一键复制为"我的出行小组"的一份可编辑真实行程，并接入持续动态分析。
- **功能点**：
  - 攻略纳用：将 GuideTemplate 复制成目标小组的 Trip + ItineraryNode + Route。
  - **相对→绝对时间映射**：模板中"第 N 天/时段"按用户选择的**实际出发日期**映射为绝对 `plannedStart/End`。
  - **可修改**：套用后复用模块 6 行程编辑（改日期、删减/替换节点、调整预算）。
  - **套用后二次校验**：对照小组成员约束（日期交集、最低预算、必访地点覆盖），超限则给出提示并支持自动裁剪。
  - 套用完成后行程状态置为 PLANNED，自动纳入持续分析（见 §8.7）。
- **接口**：`POST /api/guides/{id}/apply`（body: `groupId`、`startDate`、可选覆盖项）→ 返回新 Trip。
- **实体**：复用 Trip/ItineraryNode/Route；来源 GuideTemplate；记录 `sourceGuideId` 便于溯源与统计。
- **异常/边界**：模板缺坐标/成本时降级填充；套用后与成员约束冲突的提示与裁剪；套用生成的行程与自建行程走同一套监测/重规划闭环。
- **算法**：模板实例化 + 持续分析见 §8.7。

## 8. 核心算法设计（项目难点）

### 8.1 初始方案生成（模块 5）
输入：某小组全体 MemberConstraint。
步骤：
1. **日期交集**：`from = max(各成员 availableFrom)`，`to = min(各成员 availableTo)`；若 `from > to` → 抛出冲突（无共同可用日期）。
2. **预算封顶**：`budget = min(各成员 maxBudget)`（最保守）。
3. **必访地点**：合并去重所有 `mustVisitPlaces`。
4. **节点排布**：按顺序为每个必访地点生成 ATTRACTION 节点，叠加 MEAL/LODGING 节点，沿日期窗分配时间段。
5. **预算裁剪**：累计成本超过 `budget` 时，舍弃可选（餐饮/住宿）节点，保留必访地点。
6. **路线生成**：相邻节点间生成 Route（距离/时长/费用估算）。
7. **取舍说明**：输出解释文本（日期交集、预算依据、被裁剪项）。
进阶方向：CSP/ILP 求解器（OptaPlanner / OR-Tools / Choco），多目标（成本 vs 覆盖 vs 体验）Pareto 权衡。

### 8.2 事件—行程影响匹配（模块 10）
对每个活跃 ExternalEvent 与每个 ItineraryNode：
1. **地理判定**：Haversine 计算事件中心与节点坐标球面距离，`distance ≤ event.radiusKm` 视为地理命中。
2. **时间判定**：事件时间窗 `[startTime, endTime]` 与节点 `[plannedStart, plannedEnd]` 有重叠。
3. **命中即生成 ImpactAssessment**，将节点状态置为 AFFECTED，`impactLevel` 由 severity + 重叠程度映射。
进阶：以路网可达性替代直线距离；对 Route 的影响；概率化影响。

Haversine：
```
a = sin²(Δφ/2) + cosφ1·cosφ2·sin²(Δλ/2)
d = 2R·atan2(√a, √(1−a)),  R = 6371km
```

### 8.3 风险评分（模块 11）
`riskScore(0–100) = clamp( w1·severityScore + w2·affectedNodeRatio + w3·costExposure + w4·(1 − timeBufferRatio) )`
- severityScore：LOW/MEDIUM/HIGH/CRITICAL 映射分值。
- affectedNodeRatio：受影响节点 / 总节点。
- costExposure：受影响节点成本 / 总预算。
- timeBufferRatio：距最近受影响节点开始的缓冲时间归一化。
- 权重 w1..w4 可配置。行程级取聚合（加权或最大）。

### 8.4 自动重规划（模块 12）
对存在影响评估的行程，分别按三种策略生成 AlternativePlan：
- **MIN_EXTRA_COST**：优先改期到不额外收费时段、就近低价替换，最小化增量成本。
- **MIN_DELAY**：优先保持时间，最小化总延误（必要时替换而非顺延）。
- **MIN_CHANGE**：改动节点数最少，只动必须动的受影响节点。
每套方案为受影响节点生成 NodeChange（RESCHEDULE/REPLACE/REMOVE/ADD），计算 `extraCost / extraDelayMinutes / changedNodeCount` 与说明，状态 PROPOSED 入库。
进阶：建模为"在原解邻域的最小扰动重优化"，与 §8.1 共用求解器。

### 8.5 地图路线（模块 7）
- 首选接入地图开放平台（距离矩阵 / 路线规划 API）获取真实 `distanceKm / durationMinutes / polyline`。
- 降级：API 失败时用 Haversine 直线距离 × 修正系数估算。
- 多交通方式分别取值。

### 8.6 多人投票一致性（模块 13）
- 状态机：PROPOSED →(start-voting)→ VOTING →(tally)→ ACCEPTED / REJECTED。
- 一人一票：`(planId, memberId)` 唯一，重复投票 upsert 覆盖。
- 计票：`approves > 成员总数 / 2` 判定通过。
- **原子落地**（事务内）：应用 NodeChange 到行程 → 其余同 Trip 方案置 REJECTED → 生成 ChangeLog（含 extraCost、refundDeadline=now+24h）→ 更新 Trip → WebSocket 广播。
- 并发保护：事务 + 乐观锁/版本号，tally 幂等（已 ACCEPTED 不再重复应用）。

### 8.7 攻略套用与定时持续分析（模块 19/20 + 动态闭环）
**(1) 模板实例化（套用）**
1. 读取 GuideTemplate 及其模板节点（相对表示：`dayOffset` + `startTime`）。
2. 以用户选择的 `startDate` 为基准：`plannedStart = startDate + dayOffset 天 + startTime`，得到每个节点的绝对时间。
3. 复制坐标、类型、成本、顺序，生成目标小组的 Trip + ItineraryNode + Route。
4. 对照成员约束二次校验（日期交集/最低预算/必访覆盖），冲突则提示并可自动裁剪；行程状态置 PLANNED。

**(2) 随时间持续分析（关键：为什么分析结果会随时间变化）**
- 套用生成的行程拥有**绝对时间**后，与自建行程一样纳入监测闭环，不做特殊处理。
- **只评估未发生的节点**：影响匹配（§8.2）以"当前时间 `now`"为界，`plannedEnd < now` 的已过节点不再预警，随时间推进被评估的节点集合不断变化。
- **风险随时间上升**：风险评分（§8.3）含 `timeBufferRatio = 距节点开始的剩余时间归一化`。同一事件，越临近节点开始，`(1 − timeBufferRatio)` 越大，风险分越高——因此分数会随时间自然变化。
- **周期性重跑**：用 `@Scheduled` 定时任务（如每 N 分钟）对进行中的行程重跑 `assess → risk`；发现新的中/高影响时自动触发重规划（§8.4）并 WebSocket 推送，回到"监测→分析→重规划→投票→落地"循环。
- **触发式补充**：新事件入库（模块 9）时，也即时对相关行程触发一次增量分析，保证实时性。

```
套用/自建行程(绝对时间)
        │
        ▼
@Scheduled 定时 + 新事件触发
        │
        ▼
仅未来节点 → 影响匹配(随 now 变化) → 风险(随剩余时间上升)
        │ 有新中/高影响
        ▼
自动重规划 → 推送 → 投票 → 落地 →(继续监测)
```

### 12.3 建议实现顺序（3 人协作）
1. **第 1 步（共同打地基）**：数据模型 + 实体/DTO 契约 + 统一异常 + 认证(1)。三人先对齐实体与接口，避免返工。
2. **第 2 步（并行铺开）**：
   - A：初始规划(5) → 行程/节点数据就绪

   - C：好友(2)、约束录入(4)、行程编辑(6)
3. **第 3 步（核心闭环）**：A 做影响匹配(10) → 风险(11) → 重规划(12)；B 做投票(13) + 变更(14)。跑通"抗中断"主线。

6. **第 6 步（增值与收尾）**：C 做 AA 分账(16)、讨论区(17)、统计后台(18)；三人联调、测试、部署、文档。

> 关键路径（决定项目技术含量的主链，优先保障）：`数据模型/认证 → 5 → 10 → 11 → 12 → 13 → 20`。
