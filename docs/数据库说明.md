# 智能旅游平台数据库说明

## 1. 数据库概况

演示数据库名称为 `trip_adaptive`，使用 MySQL 8、InnoDB 和 `utf8mb4` 字符集，排序规则为
`utf8mb4_unicode_ci`。数据库保存账号、好友与组队、行程规划、监测与应变、社区攻略以及
后台统计所需的业务数据。

本次初始化包含：

- 23 张 InnoDB 业务表；
- 5 个查询视图；
- 一套可完整串联“杭州春日多人出行—事件影响—替代方案—投票采纳—通知”的演示数据。

概念关系图可参考：[核心实体 E-R 图](uml/ER图_核心实体.png)。核心关系为：`User` 通过
`GroupMember` 加入 `TravelGroup`，成员约束生成 `Trip`、`ItineraryNode` 和 `Route`；
`ExternalEvent` 通过 `ImpactAssessment` 影响行程节点，进一步产生 `AlternativePlan`、
`NodeChange`、`PlanVote` 和 `ChangeLog`；`TravelGuide` 及模板节点可以作为行程来源。

## 2. 表结构说明

下表列出每张表的用途、关键字段、主外键、索引和逻辑删除策略。金额字段统一使用
`DECIMAL(12,2)`，地理坐标使用 `DOUBLE`，时间字段使用 `DATETIME(3)`。

| 表名 | 用途 | 关键字段及约束 | 主键 / 外键 | 主要索引与逻辑删除 |
|---|---|---|---|---|
| `users` | 平台用户、角色和账号状态 | `id BIGINT UNSIGNED`；`email VARCHAR(190)` 唯一；`password_hash VARCHAR(255)`；`role`、`status`；`deleted_at` | PK `id` | `uk_users_email`、`idx_users_name`、`idx_users_status`；使用 `deleted_at` 逻辑删除 |
| `friendships` | 用户好友申请和关系状态 | `requester_id`、`addressee_id`、`pair_key VARCHAR(43)` 唯一、`status` | PK `id`；FK 两个用户字段指向 `users.id` | `uk_friendships_pair`、申请人/接收人状态索引；事实关系不设逻辑删除 |
| `travel_groups` | 出行小组和房间码 | `owner_user_id`、`room_code VARCHAR(16)` 唯一、`room_code_expire_at`、`member_limit`、`status`、`deleted_at` | PK `id`；FK `owner_user_id` → `users.id` | `uk_travel_groups_room_code`、`idx_travel_groups_owner`；使用 `deleted_at` |
| `group_members` | 用户与小组的成员关系及角色 | `group_id`、`user_id` 组合唯一；`role`（OWNER/MEMBER）；`status`；`joined_at`、`left_at` | PK `id`；FK `group_id` → `travel_groups.id`、`user_id` → `users.id` | `uk_group_members_user`、`idx_group_members_user`；通过 `status`/`left_at` 表示离组 |
| `member_constraints` | 成员日期、预算、地点、饮食和无障碍约束 | `member_id` 唯一；`available_from/to DATE`；`max_budget DECIMAL(12,2)`；三类 JSON 约束 | PK `id`；FK `member_id` → `group_members.id` | `uk_member_constraints_member`；约束记录不使用逻辑删除 |
| `trips` | 小组行程及预算快照 | `group_id`；`start_date/end_date`；`total_budget`、`spent_budget`；`source_guide_id`；`version`；`deleted_at` | PK `id`；FK `group_id` → `travel_groups.id`；`source_guide_id` 在攻略表创建后加 FK → `travel_guides.id` | 小组状态、时间状态、来源攻略索引；使用 `deleted_at` |
| `itinerary_nodes` | 行程中的景点、餐饮、住宿等节点 | `trip_id`；坐标 `latitude/longitude DOUBLE`；时间窗；`cost DECIMAL(12,2)`；`sequence_order` 组合唯一；`deleted_at` | PK `id`；FK `trip_id` → `trips.id` | `uk_nodes_trip_sequence`、时间/状态/地理索引；使用 `deleted_at` |
| `routes` | 相邻行程节点间的路线 | `from_node_id`、`to_node_id`；`transport_mode`；`distance_km DOUBLE`；`duration_minutes`；`cost`；`source` | PK `id`；FK `trip_id`、起点和终点节点 | `uk_routes_trip_nodes`、`idx_routes_trip`；路线事实不使用逻辑删除 |
| `external_events` | 天气、交通和文旅事件 | 事件类型、严重度、地点坐标、有效时间窗、来源和 JSON 原文 | PK `id`；无业务外键 | 活跃时间、类型严重度、地理时间组合索引；通过 `status` 管理生命周期 |
| `impact_assessments` | 事件与行程节点的影响匹配及风险结果 | `trip_id`、`event_id`、`affected_node_id` 组合唯一；`risk_score TINYINT`；`impact_level`；`assessment_hash` | PK `id`；FK 行程、事件、受影响节点 | `uk_impact_trip_event_node`、行程风险、事件和节点索引；评估事实不逻辑删除 |
| `alternative_plans` | 自动重规划产生的候选方案 | `strategy`；额外费用/延误/变更节点数；`status`；`source_assessment_hash` | PK `id`；FK `trip_id` → `trips.id` | 行程状态、策略哈希索引；候选方案通过 `status` 管理 |
| `node_changes` | 候选方案对节点的替换、改期和删除变更 | `change_type`；原节点；新地点、时间、坐标、费用和说明 | PK `id`；FK `plan_id` → `alternative_plans.id`、`original_node_id` → `itinerary_nodes.id` | `idx_node_changes_plan`；变更事实不逻辑删除 |
| `plan_votes` | 成员对替代方案的协同投票 | `plan_id`、`member_id` 组合唯一；`choice`；`comment`；`voted_at` | PK `id`；FK 方案和小组成员 | `uk_plan_votes_plan_member`、方案选项索引；投票事实不逻辑删除 |
| `change_logs` | 已采纳方案和行程变更审计 | `trip_id`；说明；额外费用；退款截止时间；关联方案 | PK `id`；FK `trip_id` → `trips.id`、`related_plan_id` → `alternative_plans.id` | `idx_change_logs_trip_created`；审计事实不逻辑删除 |
| `bills` | 行程费用和付款人 | `trip_id`；`amount DECIMAL(12,2)`；分类；`paid_by_member_id`；分摊模式 | PK `id`；FK 行程和付款成员 | `idx_bills_trip_created`；费用事实不逻辑删除 |
| `bill_shares` | 费用按成员的金额或比例分摊 | `bill_id`、`member_id` 组合唯一；`share_amount`；`share_ratio` | PK `id`；FK 账单和成员 | `uk_bill_shares_bill_member`；分摊事实不逻辑删除 |
| `comments` | 行程/方案讨论、回复和提及 | `target_type`、`target_id`；`parent_id` 自引用；`mentions JSON`；`like_count`；`deleted_at` | PK `id`；FK `author_id` → `users.id`、`parent_id` → `comments.id` | 目标和作者索引；使用 `deleted_at` 软删除评论 |
| `notifications` | 事件、影响、方案、投票和采纳通知 | `user_id`；`type`；标题、详情和 `payload JSON`；`read_at` | PK `id`；FK `user_id` → `users.id` | `idx_notifications_user_read`；通知为事实记录，不物理删除 |
| `travel_guides` | 社区攻略和攻略热度统计 | `author_id`；城市、主题、天数、预算；`rating`、收藏/浏览计数；`status`；`deleted_at` | PK `id`；FK `author_id` → `users.id` | 多条件筛选、评分、热度和 FULLTEXT 索引；使用 `deleted_at` |
| `guide_templates` | 攻略可套用的相对时间模板 | `guide_id` 唯一；`total_days`；模板标题 | PK `id`；FK `guide_id` → `travel_guides.id` | `uk_guide_templates_guide`；随攻略状态管理 |
| `guide_template_nodes` | 模板中的相对日期、时间和地点节点 | `template_id`；`day_offset`；`start_time/end_time`；节点顺序组合唯一；费用和备注 | PK `id`；FK `template_id` → `guide_templates.id` | `uk_template_nodes_order`、模板日期索引；模板节点不单独逻辑删除 |
| `guide_favorites` | 用户收藏攻略 | `guide_id`、`user_id` 组合唯一 | PK `id`；FK 攻略和用户 | `uk_guide_favorites_guide_user`、用户索引；收藏事实不逻辑删除 |
| `guide_ratings` | 用户对攻略评分和评论 | `guide_id`、`user_id` 组合唯一；`rating TINYINT`；评论 | PK `id`；FK 攻略和用户 | `uk_guide_ratings_guide_user`、攻略评分索引；评分事实不逻辑删除 |

### 2.1 外键顺序和 DDL 修正

`trips.source_guide_id` 依赖后面创建的 `travel_guides`，因此 `schema.sql` 先建立可空字段和
索引，所有攻略表创建完成后再添加 `fk_trips_source_guide`。源文档使用
`utf8mb4_0900_ai_ci`，本次按任务要求统一调整为 MySQL 8 可用且兼容性更好的
`utf8mb4_unicode_ci`；金额、坐标、房间码唯一约束、投票组合唯一约束和逻辑删除字段均保留。

## 3. 视图说明

| 视图 | 用途与主要字段 | 依赖表 | 示例查询 |
|---|---|---|---|
| `v_trip_overview` | 行程概览：行程、小组、房间码、成员数、节点数、节点计划成本、总预算、已用预算和剩余预算 | `trips`、`travel_groups`、`group_members`、`itinerary_nodes` | `SELECT * FROM v_trip_overview WHERE trip_id = 1;` |
| `v_trip_budget_summary` | 按行程和成员汇总分摊金额、关联账单数，并附带行程预算快照 | `trips`、`group_members`、`users`、`bills`、`bill_shares` | `SELECT * FROM v_trip_budget_summary WHERE trip_id = 1 ORDER BY member_id;` |
| `v_active_risk_snapshot` | 当前仍有效事件对应的行程最高风险、受影响节点数、事件数和最近评估时间 | `trips`、`impact_assessments`、`external_events` | `SELECT * FROM v_active_risk_snapshot ORDER BY highest_risk_score DESC;` |
| `v_guide_hot_rank` | 按浏览、收藏、评分和评分人数计算攻略热度，返回热门排序 | `travel_guides`、`guide_favorites`、`guide_ratings` | `SELECT guide_id,title,hot_score FROM v_guide_hot_rank LIMIT 10;` |
| `v_plan_vote_progress` | 替代方案投票总数、赞成/反对/弃权数、成员总数、赞成率和投票状态 | `alternative_plans`、`plan_votes`、`trips`、`group_members` | `SELECT * FROM v_plan_vote_progress WHERE trip_id = 1;` |

视图是只读查询抽象，不改变底层表的外键和事务语义。`v_plan_vote_progress` 的
`PASSED` 判断使用“赞成票超过有效成员数一半”的规则，与协同投票设计一致。

连接信息只在执行命令时提供，不能写入脚本或提交文件。使用占位符：

```bash
mysql -h <HOST> -P 3306 -u <USER> -p'<PASS>' --default-character-set=utf8mb4 \
  < docs/sql/schema.sql
mysql -h <HOST> -P 3306 -u <USER> -p'<PASS>' --default-character-set=utf8mb4 \
  < docs/sql/views.sql
mysql -h <HOST> -P 3306 -u <USER> -p'<PASS>' --default-character-set=utf8mb4 \
  < docs/sql/seed.sql
```

### 5.2 验证命令

```sql
USE trip_adaptive;
SHOW TABLES;

SELECT * FROM v_trip_overview LIMIT 5;
```

外键引用一致；视图均可查询。
