# 智能旅游平台设计过程与数据流说明

> 本文面向课程设计评审、答辩和后续联调，说明智能旅游平台如何从需求建模推进到页面、接口、业务服务、持久化和实时通知，并用可追溯的数据流串起各层。本文描述的是 Spring Boot 3 + MyBatis + MySQL 8 + Redis + JWT + WebSocket/STOMP 目标后端；前端已经提供 React 18 + Vite + TypeScript + Tailwind 的 Mock/真实接口双模式。

## 1. 设计目标与总体架构

### 1.1 设计目标

智能旅游平台不是只展示静态路线的旅游信息页面，而是围绕“多人共同规划—外部事件影响—团队协商—路线恢复”建立闭环。设计目标如下：

1. **让多人约束可以落到可执行方案。** 将成员日期、预算、必访地点、饮食和无障碍需求保存为结构化数据，生成方案时可以解释每个取舍。
2. **让突发事件可以被追溯。** 外部天气、交通、景区和大型活动事件进入系统后，依据地点距离和时间窗匹配到具体行程节点。
3. **让风险评估可以解释。** 风险结果不仅有 0—100 分，还返回事件严重度、受影响节点占比、成本暴露和时间缓冲等因子。
4. **让变更由团队共同决定。** 系统生成低额外成本、最少延误、最少改动三类方案，通过投票状态机和事务保证最终应用一致。
5. **让前后端可以渐进式联调。** 前端默认 Mock 数据即可独立演示，设置 `VITE_USE_MOCKS=false` 后切换到 `VITE_API_BASE` 指向的真实 REST 服务；真实后端接入 WebSocket 后再补上实时事件。
6. **让每个设计决定有依据。** 用例表明确“涉及的文档和数据”，再由领域模型、数据库表、DTO 和页面逐层展开，避免类设计与数据设计脱节。

### 1.2 技术栈与边界

| 层次 | 技术/组件 | 本文关注点 |
|---|---|---|
| 浏览器前端 | React 18、React Router、TypeScript、Vite、Tailwind CSS | 页面状态、表单、路由、Mock/真实 API、TanStack Query 缓存 |
| 前端数据层 | `src/api/client.ts`、TanStack Query | `VITE_USE_MOCKS` 判定、请求封装、加载/失败/空态 |
| REST 接入层 | Spring Boot 3 Controller | JWT 过滤、参数校验、`Result<T>`、HTTP 状态码 |
| 实时接入层 | WebSocket/STOMP `/ws` | 订阅 `/topic/trips/{tripId}`，接收事件、风险、方案和投票消息 |
| 业务层 | Service、领域规则、算法、事务 | 权限、日期交集、Haversine、风险评分、重规划、投票 |
| 持久化层 | MyBatis Mapper/XML、MySQL 8 | 显式列查询、事务写入、索引、逻辑删除、视图 |
| 缓存与消息 | Redis、Redis Pub/Sub | JWT 会话、缓存、锁、幂等键、跨实例实时广播 |

前端只依赖后端公开的 HTTP/WS 地址，不保存 JWT secret，也不直接连接 MySQL 或 Redis。Controller 是前后端边界；Service 是业务规则的唯一权威位置；Mapper 是数据库访问边界；MySQL 是业务事实记录的最终来源，Redis 只保存可失效的会话、缓存、锁和消息中转状态。

### 1.4 部署形态与运行边界

开发环境可以使用一个 Vite 开发服务器和一个 Spring Boot 服务：

```text
浏览器 → http://localhost:5173
       → VITE_API_BASE=http://localhost:8080
       → Spring Boot REST /ws
       → MySQL 8、Redis
```

生产环境建议将 Vite 构建产物交给 Nginx 或静态资源服务，Spring Boot 以无状态实例运行，MySQL 与 Redis 使用独立服务。JWT 让 REST 实例无需保存本地会话；Redis Pub/Sub 让多个实例可以把同一个行程事件广播给各自连接的 STOMP 客户端。

## 2. 设计过程与方法

### 2.1 从需求到数据流的推进链

设计不是先写页面再补数据库，而是按以下链路逐步收敛：

```text
需求目标
  ↓
六大模块与角色边界
  ↓
用例、基本事件流、异常事件流
  ↓
领域实体与状态机
  ↓
数据库表、主外键、索引、视图
  ↓
REST/WS DTO 与错误契约
  ↓
Service/Mapper/Redis 实现蓝图
  ↓
React 页面、Query 状态、交互反馈
  ↓
端到端数据流和联调验证
```

### 2.2 每一步的产物和依据

| 阶段 | 主要问题 | 产物 | 对下一阶段的约束 |
|---|---|---|---|

## 3. 前后端交互机制

### 3.1 Mock/真实接口双模式

前端环境变量位于 `frontend/.env.example`，典型配置如下：

```dotenv
VITE_USE_MOCKS=true
VITE_API_BASE=http://localhost:8080
```

设置 `VITE_USE_MOCKS=false` 后，`frontend/src/api/client.ts` 中的：

```ts
const useMocks = import.meta.env.VITE_USE_MOCKS !== "false";
```

决定调用路径：

1. `useMocks === true`：`api.dashboard()`、`api.guides()`、`api.trip()` 等方法直接读取 `src/mocks/data.ts`，页面仍然执行加载、成功、空态和错误分支的渲染逻辑。
2. `useMocks === false`：真实方法通过 `request<T>()` 请求 `${VITE_API_BASE}${path}`，当前基础调用包括 `GET /api/trips`、`GET /api/users`、`GET /api/trips/{id}`、`GET /api/guides`。
3. 后端扩展 PASS 2 接口后，可为 `groups`、`events`、`plans`、`votes` 等 adapter 添加真实实现，而不改变页面组件消费的类型。
4. Mock 和真实数据必须保持相同的 `User`、`Trip`、`ItineraryNode`、`TravelGuide` 等 TypeScript 结构；数据库 snake_case 不能直接泄露到页面。

### 3.2 REST 请求生命周期

以行程详情为例：

1. React Router 匹配 `/trips/1`，`TripDetailPage` 使用 `useQuery({ queryKey: ["trip", "1"], queryFn: () => api.trip(1) })`。
2. TanStack Query 先检查内存缓存；缓存命中时直接渲染，缓存缺失时将页面置于 `LoadingState`。
3. 真实模式下 `request<Trip>("/api/trips/1")` 发送 `Content-Type: application/json` 和 JWT。
4. Controller 解析路径参数、认证用户和权限，调用 `TripService.getDetail(1, currentUserId)`。
5. Service 校验当前用户是否属于行程小组，调用 `TripMapper`、`ItineraryNodeMapper`、`RouteMapper` 批量查询。
6. Converter 将数据库对象转换为 `Trip` DTO，按 `Result<Trip>` 返回并携带 `traceId`。
7. `request<T>()` 解析成功数据，TanStack Query 写入 `["trip", "1"]` 缓存，组件渲染路线、预算和风险。
8. 401/403/404/409/500 分别进入认证、权限、资源不存在、业务冲突或通用错误路径；网络错误显示后端地址和重试入口。

### 3.3 统一响应、错误码和鉴权

目标响应信封：

```json
{
  "success": true,
  "code": "OK",
  "message": "ok",
  "data": {},
  "traceId": "a2f7..."
}
```

失败响应：

```json
{
  "success": false,
  "code": "DATE_CONFLICT",
  "message": "成员可用日期没有交集",
  "data": null,
  "traceId": "a2f7..."
}
```

| HTTP 状态 | 典型 code | 前端处理方向 |
|---:|---|---|
| 400 | `INVALID_ARGUMENT` | 标记字段错误，保留用户输入 |
| 401 | `UNAUTHENTICATED`、`TOKEN_EXPIRED` | 清理本地 token，跳转 `/login` |
| 403 | `FORBIDDEN` | 显示无权限说明，不重试相同请求 |
| 404 | `NOT_FOUND` | 显示资源不存在的空态 |
| 409 | `DATE_CONFLICT`、`ROOM_FULL`、`VOTE_STATE_CONFLICT` | 显示可行动的冲突信息，刷新相关 Query |
| 422 | `VALIDATION_FAILED` | 展示请求字段或业务校验错误 |
| 500 | `INTERNAL_ERROR` | 显示重试和返回首页入口，并记录 `traceId` |

登录流程采用 JWT + Redis 会话：

1. `POST /api/auth/login` 校验邮箱和 BCrypt 密码。
2. Service 签发只包含 `sub=userId`、`roles`、`jti`、过期时间的 JWT。
3. Redis 写入 `auth:token:{jti} → userId`，TTL 与 JWT 一致。
4. 真实 REST 请求携带 `Authorization: Bearer <JWT>`。
5. Security Filter 验证签名、过期时间、Redis 会话存在性和角色；Service 再做资源归属校验。
6. `POST /api/auth/logout` 删除 Redis token；同一 JWT 随后请求得到 401。

### 3.4 TanStack Query 缓存和失效

页面以稳定的业务键组织 Query：

| Query key | 数据 | 写操作后的失效动作 |
|---|---|---|
| `["dashboard"]` | 首页用户、行程、事件、通知 | 行程、事件、通知变更后刷新 |
| `["trip", tripId]` | 行程详情、节点、路线、风险 | 节点、方案应用、预算写入后失效 |
| `["guides"]` | 攻略列表 | 发布、收藏、评分后失效或局部更新 |
| `["guides", guideId]` | 攻略详情和模板节点 | 编辑、收藏、评分后失效 |
| `["groups", groupId]` | 小组和成员 | 加入、移除、约束修改后失效 |
| `["events"]`、`["impacts", tripId]` | 事件和影响评估 | 新事件或重新 assess 后失效 |
| `["plans", tripId]`、`["votes", planId]` | 方案和投票 | 生成方案、投票、计票后失效 |

读取状态按 `LoadingState → 成功内容/EmptyState → ErrorState` 渲染；写操作成功后使用 `invalidateQueries`，不让页面长期显示旧的预算、风险或投票结果。

### 3.5 WebSocket/STOMP 与 Redis Pub/Sub

连接和订阅约定：

```text
连接：ws://localhost:8080/ws
订阅：/topic/trips/{tripId}
```

消息示例：

```json
{
  "type": "impact-assessed",
  "tripId": 1,
  "payload": {
    "riskScore": 68,
    "affectedNodeIds": [3]
  },
  "occurredAt": "2025-04-19T09:12:00+08:00"
}
```

固定消息类型：

```text
event-ingested
impact-assessed
plan-proposed
vote-updated
plan-accepted
```

实时链路如下：

```text
事件/投票 REST 写入
  → Service 事务提交
  → 发布 Redis channel ws:trip:{tripId}
  → 每个 Spring Boot 实例订阅 channel
  → 实例向本机 STOMP session 广播
  → 浏览器收到 message
  → 更新 Query cache / invalidateQueries
  → Dashboard、TripDetail、Events、Votes 重新渲染
```

Redis Pub/Sub 只负责实时传播，不替代 MySQL 事实记录。浏览器断线重连后必须通过 REST 重新读取行程、风险和投票，避免只依赖错过的消息。

## 4. 关键数据流

以下每条数据流均按“页面操作 → 接口 → Service → Mapper/表 → Redis → 返回页面”的顺序描述。图中的接口名、DTO、表名和字段名以 [后端实现全流程](后端实现全流程.md)、[前后端说明](前端说明与前后端对接.md) 和 [持久化设计](持久化设计.md) 为准。

### 4.1 注册/登录鉴权

对应图：[账号管理时序图](uml/时序图_账号管理.png)、[账号管理活动图](uml/活动图_账号管理.png)。

#### 注册数据流

1. `/register` 页面收集 `RegisterRequest` 的 `name`、`email`、`phone?`、`password`、`avatar?`；前端先检查必填项和格式。
2. `POST /api/auth/register` 进入 `AuthController`，校验邮箱长度、密码规则和邮箱唯一性。
3. `AuthService.register()` 使用 BCrypt 计算密码摘要，构造 `User`，不把明文密码写入日志或数据库。
4. `UserMapper` 插入 `users`：`id`、`name`、`email`、`phone`、`password_hash`、`avatar`、`role`、`status`、`created_at`。
5. Service 签发 JWT；Redis 写入 `auth:token:{jti}`，可按需写入短 TTL 的 `auth:user:{userId}`。
6. Controller 返回 `Result<AuthResponse>`，其中包含 `user`、`token`、`expiresIn`。
7. 前端保存 token 后进入首页；Mock 模式不访问后端，而是使用演示用户数据完成相同页面跳转。

#### 登录数据流

1. `/login` 页面提交 `LoginRequest(email,password)`。
2. `AuthController` 将请求交给 `AuthService.login()`；Service 按邮箱查询 `users`，校验 `status` 和 BCrypt。
3. 密码错误返回 `401 INVALID_CREDENTIALS`，不区分邮箱不存在和密码错误，避免泄露账号存在性。
4. 成功后生成 JWT 和 `auth:token:{jti}`，返回 `UserVO`、token 和过期时间。
5. 后续 `GET /api/trips`、`GET /api/users/me` 等请求经过 JWT Filter 和 Redis 会话校验。
6. `POST /api/auth/logout` 删除 Redis 会话，前端清理 token 并回到登录页。

涉及数据：实体 `User`；表 `users`；DTO `RegisterRequest`、`LoginRequest`、`UserVO`、`AuthResponse`；Redis `auth:token:{jti}`、`auth:user:{userId}`。

### 4.2 房间码建组与加入

对应图：[社交组队时序图](uml/时序图_社交组队管理.png)、[社交组队活动图](uml/活动图_社交组队管理.png)。

#### 建组

1. `/groups` 页面点击“创建小组”，提交 `CreateGroupRequest(name,description,memberLimit?)`。
2. `POST /api/groups` 由 `GroupController` 接收，Service 校验用户已登录、名称长度和创建权限。
3. `TravelGroupService` 生成 6—8 位大写唯一房间码，写入 `travel_groups.room_code`；通过数据库唯一约束兜底碰撞。
4. 同一事务插入 `travel_groups(id,name,description,owner_user_id,room_code,member_limit,created_at)`。
5. 插入 `group_members(group_id,user_id,role=OWNER,joined_at)`，使创建者立即成为群主。
6. Redis 写入 `room:{roomCode} → groupId`，TTL 7 天；可同时写入小组详情短缓存。
7. 返回 `TravelGroupVO`；前端显示房间码并提供复制、分享和下一步补充约束入口。

#### 房间码加入

1. 成员在 `/groups` 的加入弹窗输入 `roomCode`，前端转大写并校验长度。
2. `POST /api/groups/join` 先经过 `room:join:rate:{ip}` 限流，再查 Redis `room:{roomCode}`。
3. Redis 未命中时由 `TravelGroupMapper` 回查 `travel_groups.room_code`，并回填缓存。
4. `GroupService.joinByRoomCode()` 事务内校验小组未删除、未过期、未满员，且当前用户不在 `group_members` 中。
5. 根据产品策略插入 `group_members(..., role=MEMBER)` 或写入待审核申请；重复提交使用幂等键避免重复成员。
6. 创建 `notifications` 记录通知群主，并通过 `ws:trip:{tripId}` 或小组主题提示成员状态变化。
7. 前端失效 `["groups"]` 和相关成员 Query，显示“已加入/等待群主确认”的明确结果。

涉及表：`travel_groups`（`room_code`、`owner_user_id`、`member_limit`、`deleted_at`）、`group_members`（`group_id`、`user_id`、`role`、`joined_at`）、`notifications`。涉及 Mapper：`TravelGroupMapper`、`GroupMemberMapper`。涉及 DTO：`CreateGroupRequest`、`JoinGroupRequest`、`TravelGroupVO`、`GroupMemberVO`。

### 4.3 成员约束 → 初始方案生成

对应图：[出行规划时序图](uml/时序图_出行规划管理.png)、[出行规划活动图](uml/活动图_出行规划管理.png)。

#### 约束写入

1. `/groups/:id/constraints/:memberId` 加载 `GET /api/groups/{groupId}/members/{memberId}/constraint`。
2. 成员提交 `MemberConstraintRequest`：`availableFrom`、`availableTo`、`maxBudget`、`mustVisitPlaces[]`、`fitnessLevel`、`dietaryNeeds[]`、`accessibilityNeeds[]`。
3. `ConstraintService` 校验日期顺序、预算正数、数组长度和当前用户/群主权限。
4. `MemberConstraintMapper` 更新 `member_constraints` 的 `available_from`、`available_to`、`max_budget`、`must_visit_places`、`fitness_level`、`dietary_needs`、`accessibility_needs`、`updated_at`。
5. 写成功后失效 `group:{groupId}`、`members:{groupId}` 和 `plan:preview:{groupId}` 缓存。

#### 生成方案

1. `/trips/new` 或小组详情页点击“生成初始方案”，提交 `GeneratePlanRequest(startDate?,title?)` 到 `POST /api/groups/{groupId}/plan`。
2. `PlanService` 获取 `plan:lock:group:{groupId}` 短锁，防止同一小组并行生成两套初始方案。
3. `MemberConstraintMapper` 读取全部未删除成员约束，计算：
   - `from = max(availableFrom)`
   - `to = min(availableTo)`
   - `budget = min(maxBudget)`
   - `places = stableDistinct(flatMap(mustVisitPlaces))`
4. 若 `from > to`，返回 `409 DATE_CONFLICT`，前端保留输入并提示成员调整可用日期。
5. `PlanService` 将必访地点放入候选节点，再按日期、成员最低体力等级、饮食和无障碍需求补充餐饮、住宿等可选节点。
6. `ItineraryNodeService` 为节点安排 `plannedStart`、`plannedEnd`、`sequence_order`；必访节点不可因预算裁剪。
7. 若节点成本总和超过最低预算，按“可选、低优先级、单位成本高”顺序裁剪；返回 `tradeOffExplanation`，明确被裁剪节点和原因。
8. `RouteService` 对相邻节点调用地图服务；失败时用 Haversine 距离乘交通方式修正系数，写入 `routes.source=ESTIMATED`。
9. 一个 MySQL 事务依次写入：
   - `trips(group_id,title,status,start_date,end_date,total_budget,spent_budget,destination,source_guide_id)`
   - `itinerary_nodes(trip_id,name,place_name,latitude,longitude,node_type,planned_start,planned_end,cost,sequence_order,status)`
   - `routes(trip_id,from_node_id,to_node_id,transport_mode,distance_km,duration_minutes,cost)`
10. 返回 `PlanVO`/`Trip`，前端跳转行程详情，TanStack Query 失效 `["trips"]` 和 `["trip",tripId]`。

这条数据流中的最小设计依据是：实体 `MemberConstraint`、`Trip`、`ItineraryNode`、`Route`；表 `member_constraints`、`trips`、`itinerary_nodes`、`routes`；DTO `GeneratePlanRequest`、`PlanVO`、`Trip`；Redis `plan:lock:group:{groupId}`、路线缓存 `route:{from}:{to}:{mode}`。

### 4.4 事件接入 → 影响匹配 → 风险评分 → 自动重规划 → 投票 → 变更 → 通知

这是系统的核心动态闭环。

对应图：[监测与应变时序图](uml/时序图_监测与应变管理.png)、[监测与应变活动图](uml/活动图_监测与应变管理.png)、[投票决策活动图](uml/活动图_投票决策.png)。

#### 4.4.1 事件接入

1. `/events` 页面或事件提供方提交 `ExternalEventRequest`：`eventType`、`title`、`description`、`placeName`、`latitude`、`longitude`、`radiusKm`、`severity`、`startTime`、`endTime`、`source`。
2. `POST /api/events` 进入 `EventController`，按来源限流 `event:ingest:rate:{source}`，校验时间范围、严重度和坐标。
3. `ExternalEventService` 写入 `external_events(id,event_type,title,description,place_name,latitude,longitude,radius_km,severity,start_time,end_time,source,created_at)`。
4. Redis 更新 `events:active` ZSET（score 为结束时间），事件详情写入 `event:detail:{id}` 短缓存。
5. 发布内部 `event-ingested`，找出状态为 `PLANNED/ONGOING` 且目的地可能相关的行程，异步或同步触发 assess。

#### 4.4.2 Haversine + 时间窗影响匹配

1. `POST /api/trips/{tripId}/assess` 或事件接入触发 `ImpactAssessmentService.assess(tripId,eventIds)`。
2. `ItineraryNodeMapper` 只读取未来节点：`planned_end >= now`，避免历史节点重新告警。
3. 对每个事件—节点对计算 Haversine：

```text
φ = latitude × π / 180
a = sin²((φ2-φ1)/2) + cos(φ1)cos(φ2)sin²((λ2-λ1)/2)
distance = 2 × 6371 × atan2(sqrt(a), sqrt(1-a))
hit = distance <= event.radiusKm
timeHit = event.start < node.plannedEnd && event.end > node.plannedStart
```

4. 同时满足空间命中和时间重叠才生成影响；按 `severity + overlapRatio` 映射 `MINOR/MODERATE/SEVERE`。
5. `ImpactAssessmentMapper` 使用 `(trip_id,event_id,node_id)` 唯一键或 upsert 写入 `impact_assessments`；同一节点多事件保留最高影响。
6. 命中的 `itinerary_nodes.status` 更新为 `AFFECTED`，清除 `trip:risk:{tripId}`。
7. 返回 `ImpactAssessment[]`，发布 `impact-assessed`，payload 至少包含 `riskScore`、`affectedNodeIds`。

#### 4.4.3 风险评分

1. `GET /api/trips/{tripId}/risk` 先查 `trip:risk:{tripId}`，命中则直接返回；未命中才计算。
2. 统计未来节点、受影响节点、最大事件严重度、受影响成本和最近节点时间缓冲。
3. 按约定权重计算：

```text
severity = LOW:0.15, MEDIUM:0.40, HIGH:0.70, CRITICAL:1.00
affectedRatio = affectedFutureNodes / futureNodes
costExposure = min(affectedCost / totalBudget, 1)
timeBufferRatio = clamp(hoursToNearestNode / 24, 0, 1)
score = round(100 × clamp(
  .40 × severity +
  .25 × affectedRatio +
  .15 × costExposure +
  .20 × (1 - timeBufferRatio), 0, 1))
```

4. 返回 `RiskVO(score,riskLabel,factors[],assessments[])`；Redis 写入 TTL 2 分钟。
5. Dashboard 的 `RiskGauge`、行程详情风险卡片和影响页面使用同一个风险结果，避免各页面自行计算出不同分数。

#### 4.4.4 自动重规划

1. 当新影响达到 `MODERATE` 以上，或用户在 `/impacts`、`/plans` 点击“生成替代方案”，调用 `POST /api/trips/{tripId}/replan`。
2. 以 `tripId + activeAssessmentHash` 查 `replan:dedup:{tripId}:{hash}`；相同输入在 TTL 10 分钟内直接复用结果。
3. 对每个 `AFFECTED` 节点构造候选邻域，并分别优化：
   - `MIN_EXTRA_COST`：最小 `extraCost`；
   - `MIN_DELAY`：最小 `extraDelayMinutes`；
   - `MIN_CHANGE`：最小 `changedNodeCount`。
4. 每个候选都重新检查成员日期、预算、饮食、无障碍、必访地点和相邻路线。
5. `AlternativePlanMapper` 写入 `alternative_plans`，`NodeChangeMapper` 写入 `node_changes`，初始状态为 `PROPOSED`。
6. 返回三组 `AlternativePlan` 及其 `nodeChanges`，发布 `plan-proposed`，通知行程成员。

#### 4.4.5 协同投票和过半应用

1. 群主在 `/plans` 选择方案，`POST /api/plans/{planId}/start-voting` 将状态由 `PROPOSED` 改为 `VOTING`。
2. 成员在 `/votes` 提交 `VoteRequest(memberId,choice,comment)` 到 `POST /api/plans/{planId}/votes`，携带 `Idempotency-Key`。
3. `PlanVoteService` 校验成员属于行程小组、方案状态为 `VOTING`；`plan_votes(plan_id,member_id)` 唯一约束保证重复提交为 upsert。
4. 写入 `plan_votes(plan_id,member_id,choice,comment,created_at)`，发布 `vote-updated`，前端失效投票 Query。
5. 群主请求 `POST /api/plans/{planId}/tally`；Service 获取 `lock:vote:tally:{planId}`，事务内 `SELECT ... FOR UPDATE` 锁定方案。
6. 若 `approveCount > groupMemberCount / 2`，执行：
   - 应用全部 `node_changes` 到 `itinerary_nodes`；
   - 将兄弟方案置为 `REJECTED`；
   - 写入 `change_logs(description,extra_cost,refund_deadline,related_plan_id,created_at)`；
   - 更新 `trips.version`/预算和节点状态；
   - 删除 `trip:risk:{tripId}` 与相关 Query 缓存；
   - 将当前方案置为 `ACCEPTED`。
7. 若全部成员投票但未过半，方案置为 `REJECTED`；若仍未投完，保持 `VOTING`。
8. 事务提交后发布 `plan-accepted` 或最终 `vote-updated`，Redis Pub/Sub channel 为 `ws:trip:{tripId}`。
9. 每个在线成员的 STOMP 客户端收到消息，更新投票进度、路线节点和风险卡片；离线成员下次打开页面从 `notifications` 和 MySQL 读取。

### 4.5 攻略纳用：相对时间 → 绝对时间

对应图：[社区时序图](uml/时序图_社区管理.png)、[社区活动图](uml/活动图_社区管理.png)。

1. `/guides/:id` 通过 `GET /api/guides/{guideId}` 获取 `TravelGuide` 和 `GuideTemplateNode[]`。
2. 用户点击“攻略纳用”，填写目标 `groupId`、实际 `startDate` 和可选 `overrides`。
3. `POST /api/guides/{guideId}/apply` 进入 `GuideApplyService`，校验攻略存在、目标用户属于小组、攻略作者/内容状态可用。
4. 每个模板节点用以下公式映射：

```text
plannedStart = LocalDateTime.of(startDate.plusDays(dayOffset), startTime)
plannedEnd   = LocalDateTime.of(startDate.plusDays(dayOffset), endTime)
```

5. 复制模板的 `name`、`placeName`、`latitude`、`longitude`、`nodeType`、`cost`、`sequenceOrder`，生成 `itinerary_nodes`。
6. 根据节点顺序生成 `routes`，创建 `trips.source_guide_id = guideId`，保存来源溯源。
7. 读取 `member_constraints` 校验日期交集、最低预算、必访地点覆盖、饮食和无障碍需求。
8. 如果超出约束，返回 `constraintWarnings[]` 和 `trimmedNodeIds[]`；用户确认后可裁剪非必访节点，不能静默删除必访节点。
9. 一个事务写入 `trips → itinerary_nodes → routes`，返回新 `Trip`。
10. 前端进入 `/trips/{id}`，后续行程继续进入事件监测和持续动态分析。

涉及表：`travel_guides`、`guide_templates`、`guide_template_nodes`、`trips`、`itinerary_nodes`、`routes`、`member_constraints`。涉及 DTO：`TravelGuide`、`GuideTemplateNode`、`GuideApplyRequest`、`Trip`。涉及字段：`trips.source_guide_id`、模板 `day_offset/start_time/end_time`、节点坐标和费用。

### 4.6 后台数据统计：视图读取与聚合

对应图：[后台与监控时序图](uml/时序图_后台与监控管理.png)、[后台与监控活动图](uml/活动图_后台与监控管理.png)。

1. 管理员访问 `/admin`，前端请求 `GET /api/admin/stats?from=&to=`，携带管理员 JWT。
2. `AdminController` 和 `AdminService` 双重检查 `role=ADMIN`，普通用户得到 403。
3. `AdminStatsMapper` 优先从 MySQL 统计视图和索引列读取：
   - `v_trip_overview`：行程、小组、成员数、节点数、预算使用；
   - `v_trip_budget_summary`：按行程/成员的预算和分摊；
   - `v_active_risk_snapshot`：当前风险和受影响节点；
   - `v_guide_hot_rank`：浏览、收藏、评分热度；
   - `v_plan_vote_progress`：方案投票进度。
4. Service 在视图结果之上聚合 `tripCount`、`budgetDistribution`、`eventHitRate`、`replanAcceptanceRate`、时间序列 `series[]`。
5. 高频看板结果写入 Redis `admin:stats:{date}`，TTL 按统计周期设置；数据源变化时可由定时任务刷新。
6. 返回 `AdminStatsVO`，前端渲染目的地柱状图、预算环图、事件源健康度和方案采纳率。

统计视图只负责读优化，不能替代 `trips`、`bills`、`impact_assessments`、`alternative_plans` 等事实表。数据修正仍由业务写接口完成，视图下一次查询即可反映结果。

## 6. 数据一致性与性能设计

### 6.1 事务边界

| 业务操作 | 事务范围 | 一致性要求 |
|---|---|---|
| 注册 | 插入 `users`、签发会话 | 邮箱唯一；密码摘要和用户记录不能部分成功 |
| 建组 | `travel_groups` + 初始 `group_members` | 房间码唯一，创建者必须成为群主 |
| 初始方案 | `trips` + `itinerary_nodes` + `routes` | 不能出现无节点行程或无来源节点的路线 |
| 影响评估 | `impact_assessments` + 节点状态 | 唯一事件-行程-节点关系，重复 assess 使用 upsert |
| 方案生成 | `alternative_plans` + `node_changes` | 方案和变更明细必须同时存在 |
| 投票计票 | 锁定方案、应用节点、写 `change_logs`、更新状态 | 过半应用一次，已 ACCEPTED 不得二次应用 |
| 费用录入 | `bills` + `bill_shares` + 预算更新 | 金额和分摊总额一致，使用 `DECIMAL`/分 |
| 攻略纳用 | `trips` + 节点 + 路线 | 来源攻略、绝对时间和节点顺序同时落盘 |

事务只覆盖 MySQL 权威事实写入；Redis 删除、Pub/Sub 发布放在提交成功之后，避免广播一个最终回滚的方案。若消息发送失败，事实仍可通过 REST 查询恢复，必要时由 outbox 或重试任务补发。

### 6.2 Redis 缓存、失效和锁

| Key 模式 | 用途 | 失效/过期策略 |
|---|---|---|
| `auth:token:{jti}` | JWT 会话黑名单/会话存在性 | TTL 与 JWT 一致，登出删除 |
| `auth:user:{userId}` | 用户短缓存 | 资料修改后删除 |
| `room:{roomCode}` | 房间码到小组 ID | TTL 7 天，建组/解散时删除 |
| `route:{from}:{to}:{mode}` | 路线结果 | TTL 6 小时，节点坐标或模式变化时删除 |
| `trip:risk:{tripId}` | 风险结果 | TTL 2 分钟，事件/assess/方案应用后删除 |
| `replan:dedup:{tripId}:{hash}` | 相同输入去重 | TTL 10 分钟 |
| `plan:lock:group:{groupId}` | 防止并发生成初始方案 | 短锁，超时自动释放 |
| `lock:vote:tally:{planId}` | 防止并发计票/应用 | 3 秒获取，事务完成后释放 |
| `idempotency:{userId}:friend:{targetId}` | 好友申请幂等 | 短 TTL |
| `idempotency:tally:{planId}:{requestId}` | 计票请求幂等 | 24 小时 |
| `ws:trip:{tripId}` | 跨实例实时消息 channel | Pub/Sub，不作为持久事实 |

缓存读取必须允许回源；缓存删除失败不能让接口假装写入失败，而应记录日志、重试并在下一次 TTL 后自然恢复。

### 6.3 索引、视图和查询性能

1. `users.email`、`travel_groups.room_code`、`group_members(group_id,user_id)`、`plan_votes(plan_id,member_id)`、`guide_ratings(guide_id,user_id)` 等唯一索引同时承担业务约束和快速查找。
2. `itinerary_nodes(trip_id,planned_start,planned_end)` 支持未来节点筛选；`external_events(start_time,end_time,event_type,severity)` 支持事件列表和活动窗口查询。
3. 所有 MyBatis 查询显式列名，详情页批量读取节点和路线，避免 N+1。
4. 行程详情可先读 `trips`，再批量读 `itinerary_nodes`、`routes`、风险和小组成员；前端用 Query cache 复用结果。
5. `v_trip_overview`、`v_trip_budget_summary`、`v_active_risk_snapshot`、`v_guide_hot_rank`、`v_plan_vote_progress` 将高频统计逻辑集中在数据库层，避免每个页面复制聚合算法。
6. 管理后台高频时间序列可由定时任务预聚合到 `admin:stats:{date}`；视图仍作为可追溯的回源来源。
7. Haversine 只对候选事件和未来节点计算，先用目的地、时间窗或范围筛选缩小计算集合；外部地图服务超时则使用可标记的估算路线。

### 6.4 限流、幂等和并发

- 房间码加入按 IP 和账号限流，防止枚举房间码。
- 外部事件按 `source` 限流，防止重复推送造成风险风暴。
- 投票写入通过数据库唯一键和 `Idempotency-Key` 双重保证重复提交不会产生多票。
- 方案计票通过 Redis 短锁 + MySQL `SELECT FOR UPDATE`，锁解决跨实例并发，事务解决最终一致。
- 自动分析通过 `analysis:dedup:{tripId}:{hash}` 避免定时任务和事件触发同时生成相同方案。
- 业务 Service 每次校验资源归属，不能只依赖前端隐藏按钮或 Controller 路径。

### 6.5 时间、金额和数据格式

- 后端接口返回 ISO-8601 时间，例如 `2025-04-19T14:30:00+08:00`；MySQL 使用统一时区存储，前端负责本地化显示。
- 金额使用 `DECIMAL(12,2)` 或以分为单位的整数，禁止使用 Double 直接累加。
- 坐标使用 `DOUBLE`；缺失坐标时必须显式标记估算或无法匹配，不能假定 `0,0` 是真实位置。
- JSON 约束字段只保存成员偏好等结构化集合；查询和统计仍需在 Service 层转换为 DTO。
- API DTO 使用 camelCase，如 `plannedStart`、`sourceGuideId`；数据库列名可以是 `planned_start`、`source_guide_id`，由 MyBatis/Jackson 显式转换。

## 7. 可追溯联调清单

评审或答辩演示可以按以下顺序检查一条完整链路：

1. 用演示账号登录，验证 `users`、JWT 和 Redis 会话。
2. 创建小组并复制房间码，验证 `travel_groups`、`group_members` 和 `room:{roomCode}`。
3. 为成员填写日期、预算和必访地点，验证 `member_constraints`。
4. 生成杭州或上海行程，检查 `trips`、`itinerary_nodes`、`routes`，并说明日期交集和最低预算算法。
5. 接入天气/交通事件，查看 `external_events` 和 `impact_assessments`，说明 Haversine 与时间窗命中。
6. 打开行程详情和影响页面，说明风险缓存、风险因子和 `v_active_risk_snapshot`。
7. 生成三套替代方案，检查 `alternative_plans`、`node_changes` 和 `replan:dedup`。
8. 发起投票、重复提交同一票、计票通过，检查 `plan_votes` 唯一约束、`change_logs` 和节点状态。
9. 观察在线成员收到 `vote-updated`/`plan-accepted`，说明 Redis Pub/Sub 到 STOMP 的跨实例链路。
10. 在攻略详情执行攻略纳用，说明相对日偏移到绝对时间、约束警告和 `trips.source_guide_id` 溯源。
11. 打开后台统计，说明五个 `v_*` 视图如何为看板提供聚合数据。
