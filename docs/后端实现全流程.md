# 智能旅游平台后端实现全流程（MyBatis / MySQL / Redis）

> 本文是目标后端的实现蓝图。仓库当前已有一套 Spring Boot + JPA + H2 基线代码，但本文**不描述那套基线的持久化实现**，而是描述用户将要实现的 Spring Boot 3.3.x、Java 17、MyBatis、MySQL 8、Redis 目标架构。

## 1. 全局分层与项目骨架

### 1.1 推荐目录

```text
com.trip.adaptive
├── controller/       REST、WebSocket 入站、参数校验
├── service/          事务编排、领域规则、算法
├── mapper/           MyBatis Mapper 接口
├── domain/           持久化对象、枚举、值对象
├── dto/
│   ├── request/      入参 DTO
│   └── response/     出参 DTO
├── converter/        DO/DTO 转换
├── config/           MyBatis、Redis、Security、WebSocket、任务
├── exception/        业务异常与统一处理
├── scheduler/        持续分析、统计刷新、排行榜回写
└── support/          Result、分页、幂等、锁、当前用户
```

### 1.2 统一响应、异常与认证

所有业务 REST 接口使用：

```java
public record Result<T>(
    boolean success,
    String code,
    String message,
    T data,
    String traceId
) {}
```

分页响应建议为 `PageResult<T>`：

```json
{
  "success": true,
  "code": "OK",
  "message": "ok",
  "data": {
    "items": [],
    "page": 1,
    "pageSize": 20,
    "total": 0
  },
  "traceId": "..."
}
```

`@RestControllerAdvice` 将校验失败、资源不存在、权限不足、业务冲突、幂等冲突和未知异常统一映射为 `Result<Void>`。HTTP 状态码仍然表达协议层语义：400、401、403、404、409、422、500。

除注册、登录、健康检查外，接口携带：

```http
Authorization: Bearer <JWT>
```

JWT 中只放 `sub=userId`、`roles`、`jti` 和过期时间；Redis 保存 `auth:token:{jti}` → `userId`，TTL 与 JWT 一致。登出时删除该 key，过滤器同时校验 JWT 签名和 Redis 会话存在性。

### 1.3 MyBatis、Redis 与实时通信

- `mybatis-spring-boot-starter` + `@MapperScan("com.trip.adaptive.mapper")`。
- XML 放在 `classpath:/mapper/**/*.xml`，所有查询使用显式列名，不使用 `select *`。
- MySQL 8 使用 HikariCP、`utf8mb4`、UTC 存储 `DATETIME`，应用层统一 `ZoneId`。
- Redis 使用 `spring-data-redis` + Lettuce；需要分布式锁时可使用 Redisson，或用 `SET key value NX EX` 实现短锁。
- WebSocket 使用 STOMP `/ws`，业务主题 `/topic/trips/{tripId}`。单实例可用 Spring broker，多实例使用 Redis Pub/Sub 传播事件，再由各实例广播给本地订阅者。

### 1.4 权限边界

Service 层每次都校验资源归属，不能只依赖 Controller：

- 普通成员：读取所属小组、行程、方案、通知、评论。
- 群主：创建/解散小组、移除成员、转移群主、发起投票、计票。
- 管理员：`/api/admin/**`。
- 只有小组成员可以操作该小组的行程、事件分析和投票；只有作者可以编辑/下架自己的攻略。

### 模块 4：成员约束

**目的：** 维护日期、预算、必访地点和个人需求。

| 方法 | 路径 | 请求/响应关键字段 |
|---|---|---|
| GET | `/api/groups/{id}/members/{memberId}/constraint` | `MemberConstraint` |

### 模块 5：初始方案生成

**目的：** 从全部成员约束生成一条可执行的初始行程。

| 方法 | 路径 | 请求/响应关键字段 |
|---|---|---|
| POST | `/api/groups/{groupId}/plan` | 入参 `startDate?,title?`；返回 `{trip,tradeOffExplanation}` |

核心算法：

```text
constraints = load all active group members' constraints
from = max(availableFrom)
to = min(availableTo)
if from > to: throw DATE_CONFLICT(409)
budget = min(maxBudget)
places = stableDistinct(flatMap(mustVisitPlaces))
nodes = placeNodes(places) + optionalMealAndLodging(from, to)
nodes = schedule(nodes, from, to, fitnessLevel=minFitness)
if sum(cost) > budget:
    remove optional nodes by priority/cost until sum <= budget
routes = routeAdjacent(nodes)
persist trip, nodes, routes in one transaction
return tradeOffExplanation
```

必访地点不可被预算裁剪；可选住宿/餐饮按成本和需求裁剪。时间安排要避开成员可用窗之外。Mapper：`MemberConstraintMapper`、`TripMapper`、`ItineraryNodeMapper`、`RouteMapper`。表：`trips`、`itinerary_nodes`、`routes`。Redis：`plan:lock:group:{groupId}`，避免同组并行生成。

### 模块 10：事件—行程影响匹配

**目的：** 用时空相交找出受影响节点。

| 方法 | 路径 | 请求/响应关键字段 |
|---|---|---|
| POST | `/api/trips/{tripId}/assess` | 返回 `ImpactAssessment[]` |
| GET | `/api/trips/{tripId}/impacts` | 返回影响列表 |

只评估 `plannedEnd >= now` 的未来节点。Haversine：

```text
φ = latitude * π / 180
a = sin²((φ2-φ1)/2) + cos(φ1)cos(φ2)sin²((λ2-λ1)/2)
distance = 2 * 6371 * atan2(sqrt(a), sqrt(1-a))
hit = distance <= event.radiusKm
timeHit = event.start < node.plannedEnd && event.end > node.plannedStart
```

命中后按 `severity + overlapRatio` 映射 `MINOR/MODERATE/SEVERE`，同一节点多事件保留最严重结果；节点置为 `AFFECTED`。写入时用 `(trip_id,event_id,node_id)` 唯一键或 upsert 去重。Mapper：`ExternalEventMapper`、`ItineraryNodeMapper`、`ImpactAssessmentMapper`。Redis：`trip:risk:{tripId}` 在新事件/重新 assess 时删除。

### 模块 11：风险评分与预警

**目的：** 输出 0–100 的行程风险和可解释因子。

| 方法 | 路径 | 请求/响应关键字段 |
|---|---|---|
| GET | `/api/trips/{tripId}/risk` | `{score,riskLabel,factors[],assessments[]}` |

建议默认权重 `w1=.40,w2=.25,w3=.15,w4=.20`：

```text
severity = LOW:0.15, MEDIUM:0.40, HIGH:0.70, CRITICAL:1.00
affectedRatio = affectedFutureNodes / futureNodes
costExposure = min(affectedCost / totalBudget, 1)
timeBufferRatio = clamp(hoursToNearestNode / 24, 0, 1)
score = round(100 * clamp(
    .40 * severity +
    .25 * affectedRatio +
    .15 * costExposure +
    .20 * (1 - timeBufferRatio), 0, 1))
```

无未来影响时返回 0。返回每个因子的原始值和解释，便于前端 RiskGauge 展示。Redis：`trip:risk:{tripId}`，TTL 2 分钟；新事件、assess、方案应用后删除。

### 模块 12：自动重规划

**目的：** 对影响节点生成最低成本、最少延误、最少改动三套方案。

| 方法 | 路径 | 请求/响应关键字段 |
|---|---|---|
| POST | `/api/trips/{tripId}/replan` | 返回 `AlternativePlan[]` |
| GET | `/api/trips/{tripId}/plans` | 方案列表 |
| GET | `/api/plans/{planId}` | 方案详情，含 `nodeChanges[]` |

对每个 `AFFECTED` 节点构造候选邻域：

```text
MIN_EXTRA_COST: 先找同日免费时段/低价候选，再比较 extraCost
MIN_DELAY: 保留原时间，优先替换而不是顺延
MIN_CHANGE: 只修改必须变更的节点，减少 changedNodeCount
for candidate:
    validate time window, budget, member constraints
    calculate extraCost, delay, changedCount
choose lexicographic objective per strategy
persist plan + NodeChange(status=PROPOSED)
```

每次生成前按 `tripId + activeAssessmentHash` 查 Redis，避免同一输入重复生成。Mapper：`AlternativePlanMapper`、`NodeChangeMapper`、`ImpactAssessmentMapper`。Redis：`replan:dedup:{tripId}:{hash}` TTL 10 分钟。

### 模块 13：多人投票一致性

**目的：** 发起投票、幂等投票、过半计票并原子应用方案。

| 方法 | 路径 | 请求/响应关键字段 |
|---|---|---|
| POST | `/api/plans/{planId}/start-voting` | 返回状态为 `VOTING` 的方案 |
| POST | `/api/plans/{planId}/votes` | 入参 `memberId,choice,comment`；返回 `PlanVote` |
| GET | `/api/plans/{planId}/votes` | `PlanVote[]` |
| POST | `/api/plans/{planId}/tally` | 返回 `{status,approveCount,rejectCount,abstainCount,trip}` |

状态机：`PROPOSED → VOTING → ACCEPTED/REJECTED`。`plan_votes` 唯一 `(plan_id,member_id)`，使用 `INSERT ... ON DUPLICATE KEY UPDATE` 保证重复提交是 upsert。

计票伪代码：

```text
lock = RedisLock("vote:tally:plan:" + planId)
if !tryLock(3s): throw RETRY
begin transaction
select plan for update
if plan.status == ACCEPTED: return stored result
assert plan.status == VOTING
votes = count by choice
if approveCount > groupMemberCount / 2:
    apply every NodeChange
    reject sibling plans
    insert ChangeLog(extraCost, refundDeadline=now+24h)
    update trip version and risk cache
    plan.status = ACCEPTED
else if all members voted:
    plan.status = REJECTED
commit
publish vote-updated / plan-accepted
unlock
```

Redis：`lock:vote:tally:{planId}` 短锁、`idempotency:tally:{planId}:{requestId}` 24 小时。MySQL 事务是最终权威，已 ACCEPTED 的方案禁止二次应用。

### 模块 19：攻略社区

| POST | `/api/guides/{id}/favorite` | 返回 `favorited,favoriteCount` |
| POST | `/api/guides/{id}/rate` | `rating,comment?`；返回新评分 |

返回字段对齐前端 `TravelGuide`：`id,title,city,days,theme,price,rating,reviews,cover,tags,author,saves,description`。查询支持最新、最热、评分、收藏排序；详情查询一次性带 template nodes。Mapper：`TravelGuideMapper`、`GuideTemplateMapper`、`GuideTemplateNodeMapper`。

### 模块 20：攻略纳用与持续分析

**目的：** 将相对时间模板实例化为目标小组的真实行程，并纳入持续监测。

| 方法 | 路径 | 请求/响应关键字段 |
|---|---|---|
| POST | `/api/guides/{guideId}/apply` | 入参 `groupId,startDate,overrides?`；返回新 `Trip` |

模板节点时间映射：

```text
plannedStart = LocalDateTime.of(startDate.plusDays(dayOffset), startTime)
plannedEnd   = LocalDateTime.of(startDate.plusDays(dayOffset), endTime)
```

复制坐标、类型、成本和顺序，创建 `Trip → ItineraryNode → Route`。之后校验日期交集、最低预算和必访地点覆盖；超限时返回 `constraintWarnings[]`，前端确认后可裁剪非必访节点。写入 `source_guide_id`，行程状态为 `PLANNED`。

持续分析：

```text
@Scheduled(fixedDelayString = "${adaptive.analysis.interval-ms:300000}")
for trip in ONGOING/PLANNED:
    futureNodes = nodes where plannedEnd >= now
    assessments = assess(trip, futureNodes)
    risk = score(assessments, now)
    if new moderate/high impact and dedup key absent:
        generateAlternatives(trip)
        publish plan-proposed
```

只评估未来节点；距离节点开始越近，`1 - timeBufferRatio` 越大，风险自然上升。`tripId + assessmentHash + floor(now/interval)` 生成 `analysis:dedup:{tripId}:{hash}`，避免重复生成相同方案。新事件入库时同步触发一次增量分析。Mapper：`GuideTemplateMapper`、`TripMapper`、`ItineraryNodeMapper`、`RouteMapper`、`ImpactAssessmentMapper`、`AlternativePlanMapper`。
