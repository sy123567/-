# 持久化设计（MySQL 8 + MyBatis + Redis）

## 1. 设计原则

- 目标数据库为 MySQL 8、存储引擎 InnoDB、字符集 `utf8mb4`、排序规则 `utf8mb4_0900_ai_ci`。
- 金额统一 `DECIMAL(12,2)`，坐标使用 `DOUBLE`，时间使用 `DATETIME(3)`。
- `created_at`、`updated_at` 使用 UTC 写入；前端按用户时区格式化。
- 业务状态使用 `VARCHAR` 保存枚举值，避免数据库 ENUM 变更导致发布困难；Java 层枚举 + MyBatis TypeHandler 负责校验。
- 可恢复的业务对象使用 `deleted_at` 逻辑删除；事件、通知、评论等事实记录不物理删除。
- 这是 MyBatis/MySQL/Redis 目标设计，不是仓库当前 JPA/H2 基线的迁移脚本。正式项目应使用 Flyway/Liquibase 管理版本。

1. `trips.source_guide_id` 是模块 20 的来源溯源字段；如果需要严格外键，可在 `travel_guides` 创建后补充：

   ```sql
   ALTER TABLE trips
     ADD CONSTRAINT fk_trips_source_guide
     FOREIGN KEY (source_guide_id) REFERENCES travel_guides(id);
   ```

2. `travel_groups.room_code` 唯一；Redis 只做快速解析和限流，MySQL 唯一索引才是最终约束。
3. `friendships.pair_key` 由应用按 `min(requesterId,addresseeId) + '_' + max(...)` 生成，保证 A→B 与 B→A 不重复。
4. `comments.target_id` 是多态关联，不能用单个外键；Service 必须校验 target 是否属于当前用户可见的 Trip/Plan。
5. JSON 数组字段适合当前规模和 DTO 形态；若需要复杂查询，可拆成 `member_constraint_places`、`member_constraint_needs` 子表。

Mapper 接口只暴露领域查询，不将 SQL 拼接散落在 Service。XML 使用显式列别名，例如 `trip_id` 映射到 `TripDO.id`。

### 3.2 Nested ResultMap 与避免 N+1

行程详情推荐三种查询：

1. 一次查 Trip；
2. 一次 `WHERE trip_id = ? ORDER BY sequence_order` 查全部节点；
3. 一次 `WHERE trip_id = ?` 查全部路线；
4. Service 组装 `TripResponse`。

Trip、Node、Route 全部强行使用一个大 `resultMap` 容易产生笛卡尔积；只有小型一对多且数据量可控时使用 nested resultMap。不要在 `TripMapper.selectPage` 的每一行调用节点 Mapper。

攻略详情可使用：

- `selectGuideWithTemplate` 主查询；
- `selectTemplateNodesByTemplateIds(List<Long>)` 批量查询模板节点；
- Java 按 templateId 分组组装。

### 3.3 攻略动态 SQL、排序与分页

```xml
<select id="selectPublishedPage" resultMap="GuideResultMap">
  SELECT id, title, cover, author_id, city, tags, theme, days,
         per_capita_budget, season, rating, rating_count,
         favorite_count, view_count, description, status,
         created_at, updated_at
  FROM travel_guides
  <where>
    status = 'PUBLISHED'
    <if test="city != null and city != ''">AND city = #{city}</if>
    <if test="theme != null and theme != ''">AND theme = #{theme}</if>
    <if test="days != null">AND days = #{days}</if>
    <if test="priceMin != null">AND per_capita_budget &gt;= #{priceMin}</if>
    <if test="priceMax != null">AND per_capita_budget &lt;= #{priceMax}</if>
    <if test="ratingMin != null">AND rating &gt;= #{ratingMin}</if>
    <if test="keyword != null and keyword != ''">
      AND MATCH(title, description) AGAINST(#{keyword} IN BOOLEAN MODE)
    </if>
  </where>
  <choose>
    <when test="sort == 'HOT'">ORDER BY favorite_count DESC, view_count DESC</when>
    <when test="sort == 'RATING'">ORDER BY rating DESC, rating_count DESC</when>
    <when test="sort == 'FAVORITES'">ORDER BY favorite_count DESC</when>
    <when test="sort == 'PRICE_ASC'">ORDER BY per_capita_budget ASC</when>
    <otherwise>ORDER BY created_at DESC</otherwise>
  </choose>
  LIMIT #{offset}, #{pageSize}
</select>
```

`sort` 必须先映射为白名单枚举，不能直接把客户端字符串拼进 `ORDER BY`。可使用 PageHelper，但要统一分页参数和 count 查询；高流量榜单优先读取 Redis ZSET。

### 3.4 TypeHandler

- `EnumTypeHandler`：Java 枚举名 ↔ VARCHAR。
- `JsonListTypeHandler`：`List<String>` ↔ MySQL JSON，使用 Jackson 反序列化并对非法 JSON 返回数据异常。
- `JsonPayloadTypeHandler`：事件 payload、通知 payload、mentions。
- 时间统一 `LocalDateTimeTypeHandler`，不在 SQL 中隐式转换时区。

### 3.5 事务边界

- 创建小组 + 群主成员 + 默认约束：一个事务。
- 初始规划：Trip、Nodes、Routes：一个事务。
- 套用攻略：Trip、Nodes、Routes、sourceGuideId：一个事务。
- `VotingService.tallyAndApply`：`@Transactional`，先锁方案/行程，再应用 NodeChange、驳回兄弟方案、写 ChangeLog、更新 Trip version。
- Redis 锁只保护跨实例并发，不能替代 MySQL 事务和行锁。

## 4. Redis 设计

| 用途 | Key 模式 | 类型 | TTL | 说明 |
|---|---|---|---|---|
| 登录会话 | `auth:token:{jti}` | String(userId) | JWT 剩余时间 | JWT 黑名单/会话存在校验，登出 DEL |
| 用户短缓存 | `auth:user:{userId}` | Hash/JSON | 10 分钟 | 资料读取加速，写入后删除 |
| 房间码解析 | `room:{roomCode}` | String(groupId) | 7 天或 roomCodeExpireAt | MySQL 唯一约束为权威 |
| 加入限流 | `room:join:rate:{ip}` | String counter | 1 分钟 | 超过阈值返回 429 |
| 行程规划锁 | `lock:plan:group:{groupId}` | String NX | 30 秒 | 防止同组并行生成 |
| 热门攻略 | `guide:hot:{sort}` | ZSET(guideId, score) | 长期 | favorites/views/rating 组合得分 |
| 攻略浏览计数 | `guide:view:{guideId}` | String counter | 1 小时窗口 | INCR，定时回写 MySQL |
| 攻略收藏计数 | `guide:favorite:{guideId}` | String counter | 1 小时窗口 | 收藏成功后 INCR，异步回写 |
| 活跃事件 | `events:active` | ZSET(eventId, endEpoch) | 事件结束后 1 小时 | `ZRANGEBYSCORE` 找活跃事件 |
| 事件详情 | `event:detail:{eventId}` | JSON | 5 分钟 | 事件入库后删除 |
| 风险快照 | `trip:risk:{tripId}` | JSON | 2 分钟 | 新事件、assess、方案应用显式删除 |
| 投票锁 | `lock:vote:tally:{planId}` | String NX | 10 秒 | 计票临界区 |
| 计票幂等 | `idempotency:tally:{planId}:{requestId}` | JSON | 24 小时 | 防止重复应用 |
| 重规划去重 | `replan:dedup:{tripId}:{assessmentHash}` | JSON(planIds) | 10 分钟 | 相同影响不重复生成 |
| 定时分析去重 | `analysis:dedup:{tripId}:{hash}:{bucket}` | String | 10 分钟 | `trip + assessmentHash + 时间桶` |
| WebSocket 广播 | `ws:trip:{tripId}` | Pub/Sub | 无 | 多实例转发事件，不做持久化 |
| API 限流 | `rate:{route}:{userId}` | String counter | 1 分钟 | 事件接入、评论、投票等接口 |

### 4.1 缓存一致性

使用 cache-aside：

1. 先读缓存，miss 再查 MySQL；
2. 权威写入只写 MySQL；
3. 事务提交后删除/刷新相关缓存；
4. 所有缓存必须有 TTL，避免永久脏数据；
5. 计数器采用 Redis INCR + 定时批量回写，回写使用增量表或 `UPDATE ... SET count = count + ?`；
6. 风险、活跃事件和方案列表在事件、assess、投票应用后显式失效。

不能只缓存的内容：投票结果、Trip/Node 状态、金额账单、ChangeLog、成员权限。它们必须以 MySQL 事务结果为准。
