## 2. 代码结构约定：包 = 模块

现有后端是扁平结构 `com.trip.adaptive.{controller,service,repository,domain,dto}`。
为便于并行开发，改为 **按模块分包**，每人只改自己的包，天然减少冲突：

```
com.trip.adaptive
├── account       ← 王一飞（账号管理）
│   ├── controller
│   ├── service
│   ├── repository
│   ├── domain
│   └── dto
├── community     ← 王一飞（社区管理）
├── social        ← 祖建明（社交组队）
├── planning      ← 祖建明（出行规划）
├── monitor       ← 宋研（监测与应变）
├── admin         ← 宋研（后台与监控）
└── common        ← 三人共用：通用配置、全局异常、工具类、鉴权、基础实体
```

**约定**
- Controller 路径按模块前缀（见上表），接口互不冲突。
- 跨模块调用走 **Service 接口**，不要直接跨包访问对方的 repository/实体内部字段。
- `common`、`application.yml`、`pom.xml`、数据库 DDL 属于**共享区**：改动前在群里说一声，尽量做成独立小 PR。

## 3. 分支策略

```
main（受保护，只能通过 PR 合入，禁止直接 push）
 └── dev（集成分支，可选：先合到 dev 联调，稳定后再合 main）
      ├── feat/account-login       王一飞
      ├── feat/account-profile     王一飞
      ├── feat/community-guide      王一飞
      ├── feat/social-roomcode      祖建明
      ├── feat/planning-generate    祖建明
      ├── feat/monitor-risk         宋研
      └── feat/admin-stats          宋研
```

**命名规范**：`feat/<模块>-<功能>`；修 bug 用 `fix/<模块>-<描述>`；文档用 `docs/<描述>`。
**原则**：一个功能一条分支，做完就开 PR，别把多个功能堆在一条长命分支上。

## 5. 在 IDEA 里配置项目

### 5.1 导入
- `File → Open` 选仓库根目录（有 `pom.xml`），IDEA 自动识别为 Maven 项目并下载依赖。
- 确认 JDK：`File → Project Structure → Project`，SDK 选 **JDK 17**。

### 5.2 后端运行配置（Spring Boot）
- 打开主类 `...AdaptiveApplication`，点行号旁绿色三角 `Run`，IDEA 自动创建 Run Configuration（默认端口 8080）。
- 配数据库环境变量：`Run → Edit Configurations → Environment variables` 加：
  ```
  DB_URL=jdbc:mysql://120.53.91.26:3306/trip_adaptive?useUnicode=true&characterEncoding=utf8mb4&serverTimezone=Asia/Shanghai
  DB_USERNAME=root
  DB_PASSWORD=你的密码
  JWT_SECRET=本地随机生成的串
  ```
  （密码只放这里或本地 `.env`，不进 Git。）

### 5.3 前端运行（Vite）
- 终端进入 `frontend`：`npm install`，然后 `npm run dev`（默认 `http://localhost:5173`）。
- `frontend/.env` 两个开关：
  - `VITE_USE_MOCKS=true` → 纯前端 mock 模式，不需要后端也能跑通 UI（做界面时用）。
  - `VITE_USE_MOCKS=false` + `VITE_API_BASE=http://localhost:8080` → 连真实后端联调。
  - 改 `.env` 后要**重启** `npm run dev` 才生效。

### 5.4 Git 工具
- IDEA 右下角显示当前分支；`Git` 菜单可 Pull/Push/New Branch/Merge/Rebase。
- 常用快捷键：`Ctrl+K` 提交、`Ctrl+Shift+K` 推送、`Ctrl+T` 更新（Pull）。

## 6. 日常开发循环（每人每天）

1. **拉最新**：`Git → Pull`（同步 `main`/`dev`），保证基线最新。
2. **建/切分支**：右下角分支菜单 → `New Branch`，基于 `main` 建 `feat/<模块>-<功能>`。
3. **写代码**：只改自己模块的包；需要改 `common`/配置时先群里同步。
4. **本地自测**：
   - 后端起 Spring Boot，`http://localhost:8080/api/...` 或 Actuator `/actuator/health` 验证；
   - 前端 `npm run dev`，`VITE_USE_MOCKS=false` 指向本地后端联调；
   - 写单元测试放 `src/test/java/对应模块`，右键 `Run Tests`。
5. **提交推送**：`Ctrl+K` commit（信息写清楚：`feat(account): 实现登录签发JWT`）→ `Ctrl+Shift+K` push 到自己的 feature 分支。

## 7. 测试分层

| 层级 | 做什么 | 工具/方式 |
| --- | --- | --- |
| 单元测试 | 对自己 Service 写用例，mock 掉 repository/外部调用 | JUnit 5 + Mockito，不连真库 |
| 接口/集成测试 | 启动上下文测 Controller→Service→DB | `@SpringBootTest` + `MockMvc`；连测试库或 H2 |
| 前端联调 | 页面 → 接口 → 数据 全链路 | 前端 `VITE_USE_MOCKS=false` 指向后端 |
| UI 冒烟 | 不依赖后端验证界面 | 前端 `VITE_USE_MOCKS=true` mock 模式 |

- 每人对自己模块的核心用例（对应 SRS 的基本/异常事件流）至少各写 1~2 个测试。
- 提 PR 前必须本地 `mvn test` 通过、前端 `npm run build`、`npm run lint` 通过。

## 8. 合并流程（PR）

1. feature 完成 → GitHub 上开 **Pull Request**，base 选 `main`（或先 `dev`）。
2. PR 描述写清：做了什么、涉及哪些接口/表、如何验证。
3. 指定**另一名组员 review**（不能自己审自己），≥1 人 approve。
4. **合并前先同步 main**：
   - IDEA 里 `Git → Pull`（更新本地 main）；
   - 切到 feature 分支，`Git → Merge 'main' into current`（或 Rebase），**本地解决冲突**后 push；
   - IDEA 的三方合并工具（`Merge` 对话框）左=对方、右=自己、中=结果，选择或手动合并。
5. CI 绿 + approve 后，在 GitHub 点 **Squash and merge** 合入 `main`，删除 feature 分支。
6. 其他人及时 `Pull main` 同步最新基线。

## 9. 减少冲突的团队约定

- **不 `git add .`**：只提交自己模块相关文件，避免带上别人的改动。
- **每天至少同步一次 `main`**：分支落后越久，冲突越大。
- **共享区改动串行化**：`common`、`application.yml`、`pom.xml`、`docs/sql/*` 的改动做成单独小 PR，改前群里吱一声。
- **数据库变更版本化**：加字段用 `ALTER`，统一走 `docs/sql/` 脚本；建议引入 **Flyway**（`src/main/resources/db/migration/V1__init.sql`、`V2__xxx.sql`）保证三人库结构一致，避免各改各的。
- **接口契约先行**：跨模块依赖别人的接口时，先约定好请求/响应 DTO（可在 `docs/前端说明与前后端对接.md` 补齐），再各自实现，避免联调时对不上。
- **提交信息规范**：`type(scope): 描述`，如 `feat(planning): 初始方案生成`、`fix(monitor): 修复风险评分空指针`。

---

## 10. 冲突处理速查（IDEA）

- Pull/Merge 出现冲突 → IDEA 弹 `Conflicts` 对话框 → 点 `Merge` 打开三方对比：
  - 左侧 = 远程/对方版本，右侧 = 本地/你的版本，中间 = 合并结果；
  - 逐块点 `>>`/`<<` 采纳，或在中间栏手动编辑；
  - 全部解决后点 `Apply`，再 commit + push。
- 拿不准的冲突（对方重构了同一个方法、大范围改动）**不要乱选**，先和对应负责人沟通再合。
